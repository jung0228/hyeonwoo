웹 에이전트 해부: 브라우저와 소통하는 법, 입출력의 진실, 그리고 평가까지

실제 데이터 샘플 하나를 잡고 — 에이전트가 브라우저와 어떻게 대화하는지, LLM에 무엇이 들어가고 무엇이 나오는지, Mind2Web과 WebArena가 왜 근본적으로 다른 문제를 푸는지 — 코드 레벨에서 전부 해부한다.

Agent · Mar 6, 2026 · 28 min read

"이거까지 알아야 해?" — 그렇다

웹 에이전트 논문을 처음 읽을 때 겪는 일: "59.1% success rate"라는 숫자를 보고 넘어가지만, 그게 어떻게 측정됐는지, LLM에 정확히 무슨 텍스트가 들어갔는지, 브라우저와 어떻게 통신하는지는 잘 모른다. 그런데 이 수준을 모르면 두 가지 문제가 생긴다.

첫째, 왜 어떤 방법이 작동하고 다른 건 안 되는지 직관이 생기지 않는다. 둘째, Mind2Web 성능과 WebArena 성능이 왜 직접 비교가 안 되는지 모른다. 같은 "웹 에이전트" 벤치마크처럼 보이지만 실제로는 완전히 다른 문제다.

이 글은 그 수준까지 내려간다. 실제 입력 텍스트가 어떻게 생겼는지, Playwright 코드가 어떻게 실행되는지, 평가 스크립트가 무엇을 확인하는지 — 구체적인 예시와 함께.

웹 에이전트의 전체 통신 스택. LLM이 직접 브라우저와 대화하는 게 아니라, Playwright → CDP → Chrome 레이어를 거친다.

1. 웹 에이전트가 브라우저와 소통하는 방법

LLM이 "검색 버튼을 클릭한다"는 결정을 내려도 실제 클릭이 일어나려면 뭔가가 물리적으로 브라우저를 조작해야 한다. 그 역할을 하는 게 Playwright 또는 Puppeteer 같은 브라우저 자동화 라이브러리다.

이 라이브러리들은 내부적으로 Chrome DevTools Protocol(CDP)을 통해 Chrome 브라우저와 WebSocket으로 통신한다. CDP는 Chrome이 노출하는 저수준 API로, 마우스 이벤트 발송, DOM 조회, JavaScript 실행, 스크린샷 캡처 등을 명령할 수 있다.

실제 Playwright 코드 — 에이전트가 쓰는 것들

import asyncio
from playwright.async_api import async_playwright

async def run_agent_step(task: str):
    async with async_playwright() as p:
        browser = await p.chromium.launch(headless=True)
        page = await browser.new_page()

        # 1. 페이지 이동
        await page.goto("https://example-shop.com")

        # 2. Accessibility Tree 추출 (LLM 입력용)
        ax_tree = await page.accessibility.snapshot()

        # 3. 스크린샷 캡처 (VLM 입력용)
        screenshot = await page.screenshot(full_page=False)

        # 4. LLM 결정 → 실행 (액션 타입별)
        # click: 요소 클릭
        await page.click('[aria-label="Search"]')

        # type: 텍스트 입력 (기존 내용 지우고)
        await page.fill('#search-input', "red canvas shoes")

        # select_option: 드롭다운 선택
        await page.select_option('#size-select', "10")

        # scroll: 페이지 스크롤
        await page.mouse.wheel(0, 500)

        # hover: 마우스 오버 (드롭다운 열기)
        await page.hover(".nav-category")

        # key_press: 키보드 입력
        await page.keyboard.press("Enter")

        # go_back: 뒤로 가기
        await page.go_back()

        # JavaScript 직접 실행 (필요한 경우)
        title = await page.evaluate("document.title")

에이전트 관점에서 액션 공간은 대략 7~10개 기본 연산으로 구성된다: click, type, select, scroll, hover, goto, go_back, key_press, 그리고 태스크 완료를 선언하는 stop. 이게 LLM이 생성해야 하는 "언어"다.

CDP 레벨에서 일어나는 일. Playwright가 page.click("#btn")을 호출하면 내부적으로 CDP Input.dispatchMouseEvent 명령을 Chrome에 WebSocket으로 보낸다. 파라미터: 이벤트 타입(mousePressed/mouseReleased), X·Y 좌표(CSS 픽셀), 마우스 버튼, modifier key 비트필드(Alt=1, Ctrl=2, Meta=4, Shift=8). Chrome은 이 이벤트를 실제 사용자 클릭과 동일하게 처리한다.

2. 관찰(Observation) — 에이전트는 뭘 보는가

브라우저에서 LLM으로 "현재 상태"를 전달하는 방법이 몇 가지 있다. 어떤 방법을 쓰느냐가 시스템 전체 성능과 토큰 비용을 크게 좌우한다.

관찰 표현 방식별 토큰 비용과 사용 사례. Amazon 상품 페이지 기준 raw HTML은 약 50K 토큰으로 실용적이지 않다.

Raw HTML → Accessibility Tree 변환 예시

아래는 같은 검색 UI를 두 가지 방식으로 표현한 예시다.

Raw HTML (일부발췌, 실제는 수만 줄)
<!-- 수천 개 div 중 일부 --> <div class="nav-search-wrapper" data-csa-c-type="widget" data-csa-c-slot-id="nav-search"> <div id="nav-search-bar-form"> <div class="nav-search-scope nav-sprite"> <select name="url" class="nav-search-dropdown searchSelect" tabindex="0" aria-label="Search in"> <option value="search-alias=aps">All</option> ... </select> </div> <input id="twotabsearchtextbox" class="nav-input nav-progressive-attribute" type="text" placeholder="Search Amazon" aria-label="Search Amazon"> <input id="nav-search-submit-button" type="submit" value="Go" class="nav-input nav-progressive-attribute"> </div> </div> <!-- 이후 수천 줄 계속... -->
Accessibility Tree (Playwright snapshot)
# page.accessibility.snapshot() 결과 { "role": "WebArea", "name": "Amazon.com", "children": [ ... { "role": "combobox", "name": "Search in", "value": "All" }, { "role": "searchbox", "name": "Search Amazon", "value": "" }, { "role": "button", "name": "Go" }, ... ] } # 결과: ~80 토큰 (raw HTML은 ~800 토큰) # 전체 페이지: AX tree ~3K vs HTML ~50K

Accessibility Tree는 브라우저가 스크린 리더 같은 보조 기술을 위해 내부적으로 유지하는 구조다. 시각적 레이아웃 정보는 없지만 "인터랙티브한 요소가 무엇이 있고 어떤 역할인가"는 정확히 담겨 있다. 텍스트 기반 에이전트에게는 이 정보만으로도 충분한 경우가 많다.

Set-of-Mark (SoM) — VLM용 시각적 grounding

스크린샷만 있으면 "어디를 클릭해야 하는가"를 VLM이 특정해야 한다. 픽셀 좌표를 직접 생성하는 건 어렵다. Set-of-Mark(SoM)는 이 문제를 해결하는 영리한 방법이다: 스크린샷 위에 인터랙티브 요소마다 번호 레이블을 오버레이하고, LLM이 "3번을 클릭하라"처럼 번호로 참조하게 한다.

SoM 구현 방법 (WebVoyager 방식). JavaScript로 페이지의 모든 인터랙티브 요소(button, input, a, select 등)를 찾아 bounding box에 검은 배경 숫자 배지를 오버레이한다. 오버레이된 스크린샷을 GPT-4V에 보내면서 각 번호에 해당하는 요소 텍스트/타입 정보도 함께 텍스트로 제공한다. 번호는 시각 정보(위치, 주변 맥락)와 텍스트 정보(역할, 레이블)를 동시에 제공하는 앵커 역할을 한다.

3. 입출력 실제 데이터 샘플 — LLM과 VLM

추상적 설명은 이쯤 하고, 실제 데이터를 보자. 태스크 하나를 잡고 두 방식(텍스트 기반 LLM, 비전 기반 VLM)에서 입력과 출력이 어떻게 다른지 나란히 보여준다.

예시 태스크: "WebArena의 쇼핑몰에서 'Waffle Knit Beanie'를 검색하고 리뷰가 가장 많은 제품을 장바구니에 담아라."

방식 A — 텍스트 기반 (WebArena 스타일)

INPUT → LLM (시스템 + 유저 메시지)
## SYSTEM PROMPT You are a web browsing agent. You will be given a task and the current state of a browser. Output your next action in this format: Thought: [brief reasoning] Action: [action_type] [arguments] Available actions: - click [element_id] - type [element_id] [text] - select [element_id] [value] - scroll [direction] [amount] - goto [url] - go_back - stop [answer] ## USER MESSAGE Task: Search for 'Waffle Knit Beanie' and add the product with the most reviews to the shopping cart. URL: http://shop.webarena.xyz/ Accessibility Tree: [1] banner "One Stop Shop" [2] link "Home" [3] link "Men" [4] link "Women" [5] link "Accessories" [6] searchbox "Search entire store here..." [7] button "Search" [8] link "My Cart (0)" [9] heading "Featured Products" [10] link "Arctix Women's Essential Insulated..." [11] link "Carhartt Men's Acrylic Watch Hat..." ... Previous actions: None What is your next action?
OUTPUT ← LLM (ReAct 포맷)
Thought: I can see a search box [6] on the page. I should search for 'Waffle Knit Beanie' to find the product. Action: type [6] Waffle Knit Beanie ------- 다음 턴 ------- # [새 페이지: 검색 결과 4개 표시] Thought: I can see search results for 'Waffle Knit Beanie'. I need to find the product with the most reviews. I can see review counts: - [23] "2-Pack Waffle Knit Beanie" - ★4.3 (127) - [31] "Organic Waffle Beanie" - ★4.1 (43) - [38] "Fleece Waffle Beanie" - ★4.5 (89) Product [23] has the most reviews (127). Action: click [23] ------- 다음 턴 ------- # [상품 상세 페이지] Thought: I'm on the product page for "2-Pack Waffle Knit Beanie". I need to add it to the cart. I can see an "Add to Cart" button. Action: click [52] # "Add to Cart" 버튼 ------- 다음 턴 ------- Thought: The product was successfully added to cart. The cart now shows (1). Task is complete. Action: stop [Task completed]

방식 B — 비전 기반 (WebVoyager/SeeAct 스타일)

INPUT → VLM (GPT-4V)
## SYSTEM PROMPT Imagine you are a robot browsing the web. You will be given a task and a screenshot of the current page with numbered labels on interactive elements. Output format: Thought: [reasoning] Action: [Click [N] | Type [N] [text] | Scroll [direction] | GoBack | Stop [answer]] ## USER MESSAGE (멀티모달) Task: Search for 'Waffle Knit Beanie' and add the product with the most reviews to the cart. [Image: 스크린샷 + SoM 번호 오버레이] ┌─────────────────────────────────┐ │ One Stop Shop [8]Cart │ │ [2]Home [3]Men [4]Women [5]... │ │ ┌──────────────────────┐[7]🔍 │ │ │[6] Search store... │ │ │ └──────────────────────┘ │ │ Featured Products │ │ [10]🧢 Arctix Beanie $24.99 │ │ [11]🧤 Carhartt Hat $19.99 │ └─────────────────────────────────┘ Element labels: [6]: searchbox "Search entire store here" [7]: button "Search" [8]: link "My Cart (0)" ...
OUTPUT ← VLM
Thought: I can see the shopping site with a search box labeled [6]. To find 'Waffle Knit Beanie', I should type the search term in the search box first. Action: Type [6] Waffle Knit Beanie ------- 다음 턴 ------- # [검색 결과 스크린샷 + 새 SoM 번호] Thought: I can see search results on the page. Looking at the products: - [12]: 2-Pack Waffle Knit Beanie Shows "127 reviews" near it - [18]: Organic Waffle Beanie Shows "43 reviews" - [24]: Fleece Waffle Beanie Shows "89 reviews" Product [12] clearly has the highest review count (127). Action: Click [12] ------- 다음 턴 ------- Thought: I'm on the product detail page. I can see the "Add to Cart" button labeled [31]. I'll click it. Action: Click [31] ------- 다음 턴 ------- Thought: The cart icon now shows "(1)", indicating the item was successfully added. The task is complete. Action: Stop [Task completed successfully]

두 방식의 핵심 차이를 정리하면: 텍스트 기반은 요소 ID([6])를 Accessibility Tree에서 직접 가져와서 확실히 존재하는 요소를 참조한다. 비전 기반은 스크린샷의 SoM 번호를 참조하는데, 이 번호는 JavaScript가 실시간으로 할당하므로 턴마다 달라질 수 있다. 어느 방식이 더 좋은가는 태스크에 따라 다르다. 시각적 레이아웃이 중요한 태스크(이미지를 보고 판단해야 하는 경우)는 VLM이 유리하고, 폼 처리·텍스트 정보 위주는 텍스트 기반이 더 안정적이다.

4. LLM → VLM: 왜 전환이 일어났나

초기 웹 에이전트들(WebGPT, WebShop 등)은 텍스트만 다뤘다. 2023년부터 스크린샷을 직접 입력으로 쓰는 VLM 기반 에이전트가 등장했고, 2024년에는 이게 사실상 주류가 됐다. 이 전환의 이유는 단순하다.

항목 텍스트 기반 (LLM) 비전 기반 (VLM)
입력 DOM / Accessibility Tree / Filtered HTML 스크린샷 (+ SoM 번호 오버레이)
처리 방식 텍스트 파싱 → 요소 ID 기반 액션 이미지 이해 → 번호·좌표 기반 액션
장점 정확한 요소 참조, DOM 변경에 강함, 토큰 효율 시각 정보 활용, DOM 파싱 불필요, 사람과 유사
단점 Canvas/SVG 요소 처리 불가, 시각 레이아웃 손실 Grounding 오류(좌표 부정확), 비쌈, 동적 요소 취약
한계 상황 JavaScript로만 렌더링되는 요소, 시각적 의존 태스크 텍스트 밀도 높은 페이지, SoM 번호 너무 많을 때
대표 시스템 Mind2Web, WebArena baseline, WebRL WebVoyager, SeeAct, Agent-E, CUA, Browser Use

현재 최강 시스템들(OpenAI CUA, Google Mariner)은 둘을 병용한다. DOM/AX Tree로 구조 정보를 얻고, 스크린샷으로 시각 정보를 확인한다. 두 채널의 정보가 충돌할 때 어느 쪽을 신뢰할지는 여전히 열린 문제다.

5. 벤치마크 심층 해부 — Mind2Web vs WebArena

가장 많이 인용되는 두 벤치마크를 제대로 이해하려면 "어떻게 데이터를 모았고, 평가 스크립트가 실제로 무엇을 확인하는가"를 알아야 한다.

Mind2Web(offline)과 WebArena(online)의 평가 파이프라인 비교. 이 둘은 같은 웹 에이전트를 평가하지만 근본적으로 다른 것을 측정한다.

Mind2Web (NeurIPS 2023) — Offline 평가의 구조

Mind2Web은 Amazon Mechanical Turk를 통해 실제 사용자들이 웹을 탐색하는 과정을 녹화해 만들었다. 137개 사이트, 2,350개 태스크, 약 7,775개 액션이 수집됐다. 각 데이터 포인트는 "그 시점의 HTML 스냅샷 + 사용자가 클릭/타이핑한 정답 요소"의 쌍이다.

실제 데이터 포맷(JSON):

{
  "annotation_id": "a3f2-9c4b-...",
  "website": "flights.google.com",
  "domain": "travel",
  "confirmed_task": "Search one-way flights NYC to LAX on Dec 15",
  "actions": [
    {
      "action_uid": "step-1",
      "raw_html": "<!DOCTYPE html>...",   // 수만 토큰
      "operation": {
        "op": "CLICK",
        "value": ""
      },
      "pos_candidates": [          // 정답 요소들
        {
          "tag": "div",
          "text": "One way",
          "xpath": "//div[@data-value='2']",
          "attributes": {"role": "radio", ...}
        }
      ],
      "neg_candidates": [...],     // 오답 요소들 (훈련용)
      "action_reprs": "click [One way]"
    },
    {
      "operation": {"op": "TYPE", "value": "New York"},
      "pos_candidates": [{"aria-label": "Where from?", ...}]
      ...
    }
  ]
}

Mind2Web의 핵심 기술 기여는 이 데이터를 활용한 2단계 Ranker-Selector 아키텍처다. 페이지당 평균 1,000개 이상의 DOM 요소를 LLM에 그냥 넣으면 컨텍스트 한계를 초과한다. 그래서 먼저 소형 cross-encoder(DeBERTa-v3-base 기반)로 태스크와 각 요소의 관련성을 스코어링해서 top-k 후보를 추린다. 그 다음 GPT-3.5/4가 top-k 중 최종 요소와 액션을 선택한다.

평가 지표 4개:

  • Element Accuracy: 모델이 선택한 요소가 pos_candidates 목록에 있는가. (top-k 중 하나면 정답)
  • Operation F1: 액션 타입(CLICK/TYPE/SELECT)과 값(type 시 입력 텍스트)의 F1 점수.
  • Step Success Rate: Element Accuracy AND Operation F1 둘 다 맞아야 1점. 이 지표가 메인이다.
  • Task Success Rate: 해당 태스크의 모든 스텝이 다 성공해야 1점. 매우 엄격해서 숫자가 낮다.
Mind2Web의 핵심 한계: 오프라인 평가. 에이전트가 실제로 브라우저를 조작하지 않는다. 저장된 HTML 스냅샷을 보고 "다음에 어떤 요소를 클릭할 것인가"를 예측하는 문제다. 실제 웹 에이전트 동작에서 중요한 것들 — 이전 액션이 페이지를 어떻게 변화시켰는가, 오류 복구, 멀티스텝 상태 추적 — 을 전혀 테스트하지 않는다. 단일 스텝 예측 정확도를 측정하는 것이지, 에이전트의 실제 태스크 완료 능력을 측정하는 게 아니다.

WebArena (ICLR 2024) — Online (Docker) 평가의 구조

WebArena의 핵심 설계 결정은 하나다: 에이전트의 액션이 실제 시스템 상태를 변경하게 하고, 그 상태를 검증한다. 4개의 완전한 웹 애플리케이션을 Docker로 호스팅한다.

  • GitLab: 이슈 생성, 코드 리뷰, 프로젝트 관리
  • Reddit 유사 포럼 (Postmill): 포스트 작성, 댓글, 투표
  • 쇼핑몰 (Magento): 상품 검색, 장바구니, 주문
  • CMS (OneStopShop): 콘텐츠 관리

실제 태스크 포맷:

{
  "task_id": 204,
  "intent": "Create a new issue titled 'Fix login timeout bug' in the 'awesome-os' project",
  "sites": ["gitlab"],
  "start_url": "http://gitlab.webarena.xyz",

  // 평가 방법: 기능적 검증
  "eval": {
    "eval_types": ["program_html"],
    "reference_answers": {
      "must_include": ["Fix login timeout bug"]
    },
    // 에이전트 실행 후 이 프로그램을 실행해서 검증
    "program": "gitlab_check_issue_exists('awesome-os', 'Fix login timeout bug')"
  }
}

평가 스크립트는 에이전트가 태스크를 완료한 후 GitLab API를 호출해서 실제로 이슈가 생성됐는지 확인한다. 에이전트가 어떤 경로로 도달했든 상관없다 — 결과물이 존재하면 성공이다. 이게 Mind2Web과의 근본적 차이다.

WebArena의 평가 타입 3가지:

  • string_match: 에이전트의 최종 응답(텍스트)이 정답 문자열과 일치하는가. 정보 검색 태스크.
  • url_match: 태스크 완료 후 브라우저가 특정 URL에 있는가.
  • program_html: 검증 프로그램을 실행해서 DB 또는 페이지 상태를 확인. 가장 강력하고 신뢰도 높음.
WebArena의 강점과 약점. 강점: 실제 상태 변화를 검증하므로 에이전트가 진짜 태스크를 완료했는지 정확히 알 수 있다. 복수의 완료 경로를 허용하므로 창의적 해결도 인정된다. 약점: 각 에이전트 실행이 Docker 환경을 오염시킨다. 평가 전 환경을 초기화해야 하고, 이 리셋 과정 자체에 오류가 생기기도 한다. 또한 812개 태스크라는 규모는 세밀한 분석엔 충분하지만 모델 변별에는 한계가 있다.

WebVoyager — 실제 인터넷 + LLM-as-Judge

WebVoyager는 Docker 없이 실제 15개 웹사이트(Amazon, Google, GitHub, Reddit 등)에서 평가한다. 에이전트가 실제 인터넷을 돌아다닌다는 게 장점이자 단점이다 — 웹사이트 UI가 바뀌면 태스크가 깨진다.

평가 방법이 독특하다: 사람이 모든 결과를 검토하는 건 비용이 너무 많이 드니, GPT-4가 "에이전트의 최종 응답 + 마지막 k개 스크린샷"을 보고 성공/실패를 판정한다. 이 LLM-as-Judge 방식은 사람 평가자와의 일치율이 85.3%였다. 완벽하지 않지만 실용적이다. WebVoyager의 GPT-4V 에이전트 성공률 59.1%는 이 방식으로 측정됐다.

벤치마크 환경 태스크 수 평가 방식 재현성 신뢰도
Mind2Web 오프라인 (저장된 HTML) 2,350 Element Accuracy 완벽 단일 스텝만 평가
WebArena Docker (4개 앱) 812 기능적 검증 환경 리셋 필요 높음
WebVoyager 실제 15개 사이트 643 LLM-as-Judge 낮음 (UI 변경) 판정 오류 14.7%
VisualWebArena Docker + 시각 태스크 910 기능적 검증 환경 리셋 필요 높음

6. 주요 논문 기술 핵심 해부

Mind2Web (NeurIPS 2023) — Ranker-Selector 파이프라인

Mind2Web의 기술적 핵심은 계층적 필터링이다. 수천 개 DOM 요소를 한번에 LLM에 넣을 수 없으니 두 단계로 분리한다.

Ranker는 각 DOM 요소를 "태스크 설명 + 이전 액션 이력"과 쌍으로 만들어 cross-encoder로 관련성을 스코어링한다. top-50 정도를 추리고 LLM Selector에게 넘긴다. LLM은 50개 요소만 보면 되므로 컨텍스트 한계 문제가 해결된다. 이 아이디어는 이후 거의 모든 텍스트 기반 에이전트에 채택됐다.

재미있는 수치: 파인튜닝된 Flan-T5가 Selector로 사용될 때 Step Success Rate 52.0%, 파인튜닝 없이 GPT-4를 쓰면 41.7%. 오픈소스 파인튜닝 모델이 GPT-4 zero-shot을 이긴다.

WebVoyager (ACL 2024) — 단순한 시스템이 왜 강력한가

WebVoyager의 놀라운 점은 아키텍처의 단순함이다. Mind2Web처럼 정교한 DOM 필터링이 없다. 그냥 GPT-4V에 SoM 스크린샷을 넣고 ReAct 포맷으로 다음 액션을 물어본다. 그것만으로 59.1%가 나왔다.

이 결과가 의미하는 것: GPT-4V의 시각 이해 능력이 복잡한 DOM 파싱보다 실제 웹 탐색에 더 효과적일 수 있다. 텍스트 에이전트가 놓치는 시각적 맥락 — 버튼의 위치, 색상으로 구분되는 상태, 이미지로만 표현된 정보 — 을 자연스럽게 처리한다.

GPT-4V 비교 (WebVoyager 내 실험):

  • GPT-4 텍스트만 (모든 툴): 30.8%
  • WebVoyager 텍스트만: 40.1%
  • WebVoyager 스크린샷 포함: 59.1%

스크린샷 추가만으로 20% 포인트 향상. 시각 정보의 임팩트다.

Agent-E (arXiv 2024) — 계층적 분리가 주는 것

Agent-E는 단일 에이전트 구조 대신 Orchestrator와 Browser Navigator를 분리한다. Orchestrator는 고수준 계획만 담당하고 ("검색 결과에서 가장 저렴한 옵션을 찾아라"), Browser Navigator는 실제 브라우저 조작만 담당한다 ("searchbox에 'red shoes'를 입력하고 결과를 스크롤하라").

핵심 기술은 DOM Distillation이다. 태스크에 따라 세 가지 표현 중 하나를 선택한다.

  • text_only: 텍스트 내용 위주. 정보 검색·요약 태스크.
  • input_fields: input·button 요소 위주. 폼 작성·검색 태스크.
  • all_fields: 모든 인터랙티브 요소. 탐색이 필요한 복잡한 태스크.

각 요소에는 mmid 라는 고유 식별자를 삽입한다. LLM이 mmid=291을 참조하면 Playwright가 그 요소를 정확히 찾아 조작한다. 결과: WebVoyager 73.2%, 당시 SOTA. 텍스트 전용 대비 20%, 멀티모달 전용 대비 16% 향상. 평균 ~25번의 LLM 호출로 태스크 완료.

Browser Use (오픈소스) — 연구가 실제 도구가 되는 과정

Browser Use는 "웹 에이전트를 직접 쓰고 싶다"는 수요를 빠르게 채운 오픈소스 프레임워크다. GitHub 3개월 만에 50K+ stars. Playwright + LangChain 기반이고 OpenAI, Anthropic, Google, Ollama 등 주요 LLM을 모두 지원한다. 

from browser_use import Agent
from langchain_openai import ChatOpenAI

agent = Agent(
    task="Find the cheapest red Nike shoes on Amazon and add to cart",
    llm=ChatOpenAI(model="gpt-4o"),
)
result = await agent.run()

이 3줄 뒤에서 일어나는 일:

1
DOM 추출
Playwright로 페이지를 열고 인터랙티브 요소들을 추출한다. 단순 selector가 아니라 요소의 시각적 위치, 텍스트 내용, ARIA 속성을 통합한 구조화된 표현을 만든다.
2
스크린샷 캡처
DOM 정보만으로 부족한 시각적 맥락을 위해 스크린샷도 함께 캡처한다. DOM 구조와 시각 정보를 병합해서 LLM에 전달한다.
3
LLM 결정
태스크 + DOM 요약 + 스크린샷을 GPT-4o에 보낸다. Function calling으로 구조화된 액션을 받는다: {"action": "type", "element_id": 5, "text": "red nike shoes"}
4
실행 + 반복
Playwright로 액션을 실행하고 새 상태를 관찰한다. 태스크 완료 또는 최대 스텝 도달까지 반복한다. 이력(방문 페이지, 과거 액션)은 메모리에 저장된다.

최신 버전은 내부적으로 Playwright에서 직접 CDP로 이동하고 있다. WebSocket 통신으로 더 세밀한 브라우저 제어가 가능하고, 일부 Playwright 추상화 레이어가 제거되어 성능이 향상된다.

7. 하나의 태스크를 End-to-End로 추적하면

지금까지 개별 구성요소를 살펴봤다. 이제 실제 WebArena 스타일 태스크 하나를 처음부터 끝까지 따라가 보자.

태스크: "WebArena GitLab에서 'awesome-os' 프로젝트의 open 이슈 수를 알려라."

0
초기화
Docker GitLab 인스턴스를 초기화(리셋). Playwright 브라우저 시작. await page.goto("http://gitlab.webarena.xyz") 실행. 로그인 세션 설정.
1
첫 번째 관찰 → LLM
AX Tree 추출: [1] link "Projects" [2] searchbox "Search GitLab" [3] link "awesome-os" ...
LLM Thought: "awesome-os 프로젝트 링크가 보인다. 클릭하면 이슈 목록을 볼 수 있을 것이다."
LLM Action: click [3]
2
프로젝트 페이지 → 이슈 탭
새 AX Tree: [12] link "Issues (7)" [13] link "Merge Requests" ...
LLM Thought: "Issues 탭에 7이라는 숫자가 보인다. 하지만 이게 open 이슈 수인지 확인이 필요하다."
LLM Action: click [12]
3
이슈 목록 확인
새 AX Tree: [1] tab "Open (5)" [2] tab "Closed (2)" [3] tab "All (7)"
LLM Thought: "Open 이슈가 5개임을 확인했다."
LLM Action: stop [5]
4
평가 (기능적 검증)
평가 스크립트 실행: string_match(answer="5", reference="5") → ✓ Success
또는 더 복잡한 태스크의 경우: GitLab API 호출로 실제 DB 상태 검증.

이 4스텝 태스크는 비교적 단순했다. 실제 WebArena 태스크들은 평균 10~15스텝이고, 어려운 것들은 30스텝을 넘는다. 각 스텝마다 LLM API 호출이 발생하므로 긴 태스크는 비용도 시간도 많이 든다.

연구자가 알아야 할 수준

여기까지 읽었으면 이 수준의 질문들에 답할 수 있어야 한다:

  • 왜 Mind2Web Step Success Rate 52%와 WebArena 42%를 직접 비교하면 안 되는가?
  • Accessibility Tree가 Raw HTML보다 왜 더 좋은 LLM 입력인가, 그리고 언제 부족한가?
  • SoM이 grounding 문제를 해결하는 원리가 무엇인가, 그리고 왜 완전한 해결이 아닌가?
  • WebArena 평가에서 program_html 타입이 string_match보다 신뢰도가 높은 이유는?
  • Ranker-Selector 파이프라인이 없으면 GPT-4 기반 에이전트가 왜 많은 웹페이지에서 실패하는가?
웹 에이전트 연구에서 "성능 숫자"는 벤치마크 설계를 이해하지 않으면 의미 없다. 어떤 관찰 표현을 썼는지, 평가가 offline인지 online인지, 태스크 분포가 어떤지 — 이 맥락 없이 87%와 42%를 비교하는 건 사과와 오렌지를 비교하는 것이다.

3개의 포스트에 걸쳐 — 기초 개념, 연구 흐름 4년, 그리고 이 글 — 웹 에이전트를 여러 수준에서 봤다. 어떤 수준이 가장 유용할지는 독자에 따라 다르겠지만, 결국 세 수준을 모두 알아야 연구에서나 구현에서나 정확한 판단을 할 수 있다.
