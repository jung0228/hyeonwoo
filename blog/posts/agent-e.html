Agent-E: 웹 자동화에서 에이전틱 시스템 설계 원칙으로

계층적 아키텍처와 DOM 디노이징으로 WebVoyager 벤치마크 73.2%를 달성한 텍스트 전용 웹 에이전트.

Agent · LLM · Mar 2026 · 20 min read

먼저 한 가지 사실부터

Agent-E는 텍스트만 본다. 스크린샷을 안 쓴다. 사이트별로 특화된 프롬프트도 없다. 그런데 WebVoyager 벤치마크에서 73.2% 를 기록했다.

비교 대상을 보면 이게 얼마나 놀라운지 알 수 있다:

에이전트 타입 성공률 특이사항
WebVoyager 멀티모달 57.1% GPT-4V, 스크린샷 활용
WILBUR 텍스트 전용 52.6% 사이트별 특화 프롬프트 사용
Agent-E 텍스트 전용 73.2% 범용, 사이트 특화 없음

시각 능력도 없고 사이트별 맞춤도 없는데, 둘 다 갖춘 경쟁자들을 10~21%p 이상 앞선다. 어떻게 가능했을까?

논문 제목이 힌트를 준다. "From Autonomous Web Navigation to Foundational Design Principles in Agentic Systems" — 단순히 에이전트 하나를 만든 것이 아니라, 에이전틱 시스템을 잘 만들기 위한 보편적 원칙을 도출했다고 주장한다.

· · ·

시스템 개요: 두 에이전트의 분업

Agent-E는 두 개의 LLM 에이전트가 계층을 이루는 구조다.

Planner Agent는 사용자의 고수준 요청을 받아 서브태스크로 분해하고, 하나씩 Browser Navigation Agent에게 위임한다. 플래너는 DOM의 세부 노이즈를 직접 보지 않는다. 큰 그림만 관리하는 것이다.

Browser Navigation Agent는 각 서브태스크를 받아 실제 웹 브라우저와 상호작용한다. 결정적으로, 각 서브태스크마다 새로 초기화(freshly instantiated nested chat)된다. 이전 서브태스크의 기억이 없다. 브라우저 에이전트는 지금 당장 할 일에만 집중한다.

이 설계가 왜 효과적일까?

계층 분리의 이점
  • 플래너 오염 방지: DOM의 수천 개 요소, 에러 메시지, 상태 변화 노이즈가 플래너 컨텍스트를 오염시키지 않는다.
  • 브라우저 에이전트 집중도 유지: 이전 서브태스크 이력 없이 매번 깨끗한 상태에서 시작하므로, 지금 해야 할 일에만 집중한다.
  • 에러 격리: 한 서브태스크에서 실패해도 플래너가 감지하고 재시도 또는 대안 경로를 선택할 수 있다.

원칙 1: 도메인 특화 기본 스킬

Browser Navigation Agent가 쓸 수 있는 도구들부터 보자.

감지(Sensing) 스킬:

  • get_dom_with_content_type — DOM을 세 가지 방식으로 가져옴 (아래에서 자세히 설명)
  • get_url — 현재 페이지 URL 반환
  • get_screenshot — 스크린샷 캡처 (텍스트 에이전트도 필요시 사용 가능)

행동(Action) 스킬:

  • click — 요소 클릭 (mmid로 정밀 지정)
  • enter_text — 텍스트 입력
  • press_key_combination — 키보드 단축키
  • open_url — 새 URL 열기
  • scroll — 페이지 스크롤

논문이 강조하는 핵심:

"A well crafted ensemble of foundational skills can serve as a building block to support more complex functionalities."

스킬 목록이 단순해 보이지만, 이 스킬들이 웹 인터랙션의 원자적 단위를 정확히 커버한다. 마우스 클릭, 텍스트 입력, 키보드, URL 이동 — 인간이 웹을 쓸 때 하는 모든 행동이 여기서 조합된다.

단, 특수한 사이트에서는 오른쪽 클릭처럼 추가 스킬이 필요할 수 있다. 이 경우 도메인 특화 확장이 필요하다. 범용성과 특화의 균형이 원칙 1의 핵심이다.

· · ·

원칙 2: 페이로드 디노이징 — 세 가지 DOM 표현

Agent-E의 가장 독창적인 기여 중 하나다. get_dom_with_content_type은 단순히 DOM을 가져오는 게 아니라, 태스크 유형에 맞는 형태로 변환해서 반환한다.

세 가지 모드를 구분해서 쓰는 이유가 있다.

text_only 는 "이 페이지에서 상품 가격은?" 같은 정보 수집 태스크에 쓴다. 버튼, 입력창 같은 UI 요소는 제거하고 텍스트 콘텐츠만 뽑는다. LLM이 불필요한 인터랙티브 요소에 혼동받지 않는다.

input_fields 는 검색창에 입력하거나 폼을 작성할 때 쓴다. 상호작용 가능한 요소(<input>, <button>, <select>)만 추출하고, 각각에 mmid(mind map identifier)라는 고유 식별자를 붙인다. LLM이 "mmid=142에 클릭"이라고 지정하면, 에이전트가 정확히 그 요소를 찾아 실행한다.

all_fields 는 페이지 구조를 탐색하거나 다음 행동을 결정해야 할 때 쓴다. 전체 DOM 요소를 부모-자식 계층 구조를 보존하면서 추출한다. 경쟁 에이전트들이 평면적(flat) 인코딩을 쓰는 것과 다르다.

mmid 식별자가 왜 중요한가
기존 웹 에이전트들은 XPath나 CSS 셀렉터로 요소를 지정한다. 이 방식은 페이지가 조금만 바뀌어도 깨진다. Agent-E는 빌드 타임에 각 요소에 고유한 mmid를 주입한다. LLM이 mmid 숫자만 알면 된다 — XPath나 복잡한 CSS 선택자를 생성할 필요 없다.

이것이 원칙 3 "페이로드 디노이징" 의 핵심이다:

"Mitigating noise in the payload is critical in creating reliable, cost and time efficient agents."

상황에 따라 필요한 정보만 추출해서 LLM에게 전달하면, 판단이 정확해지고 토큰 비용도 줄어든다.

원칙 4: 언어적 행동 피드백 — 변화 관찰 메커니즘

이게 Agent-E에서 가장 흥미로운 부분이다.

일반적인 웹 에이전트는 이렇게 동작한다:

  1. DOM을 가져온다
  2. 어떤 요소를 클릭할지 결정한다
  3. 클릭한다
  4. 다시 DOM을 가져온다

즉, 자기 행동의 결과를 직접 알지 못한다. 변화를 다음 스텝의 DOM에서 간접적으로 파악해야 한다.

Agent-E는 다르다. 행동을 실행한 직후에 무슨 일이 일어났는지를 언어로 보고한다.

어떻게 구현했을까? Mutation Observer Web API를 사용한다. 브라우저 내장 API로, DOM이 변화할 때마다 실시간 콜백을 받는다. 여기에 aria-expanded 같은 접근성 속성 모니터링을 추가해, 클릭 한 번으로 어떤 변화가 일어났는지 감지한다.

논문은 이를 Reflexion과 비교한다. Reflexion은 실패 이후에 회고하는 방식이지만, Agent-E의 변화 관찰은 매 행동마다 즉각적인 환경 피드백을 제공한다. 성공/실패와 무관하게 항상 작동한다.

예를 들어 날씨 사이트에서 도시를 클릭하면:

"Clicked element mmid 48. Dropdown closed. Weather data loaded: 23°C, Sunny."

다음 행동 결정에 필요한 맥락이 즉시 주어진다. 다시 DOM 전체를 가져오지 않아도 된다.

WebVoyager 벤치마크 결과: 숫자 뜯어보기

15개 실제 웹사이트, 643개 태스크. 사이트별 성적이 흥미롭다.

웹사이트 성공률 비고
WolframAlpha 95.7% 구조적이고 예측 가능한 인터페이스
Google Search 90.7% 단순하고 일관된 UI
GitHub 82.9%
ESPN 77.3%
Apple 74.4%
Allrecipes 71.1%
Amazon 70.7%
Booking.com 27.3% 동적 달력 UI, 날짜 선택 복잡도 높음

Booking.com의 27.3%가 눈에 띈다. 여행 예약 사이트의 날짜 선택기, 인원 선택기, 동적으로 갱신되는 가격 표시 등은 텍스트 전용 에이전트에게 특히 어렵다. 클릭 한 번에 달력 UI가 열리고, 월 이동 버튼을 여러 번 눌러야 원하는 날짜에 도달하는 복잡한 다단계 인터랙션이 필요하기 때문이다.

반면 WolframAlpha처럼 검색창에 입력하고 결과를 읽는 구조적이고 예측 가능한 사이트에서는 95.7%를 기록했다.

운영 지표: 에이전트가 얼마나 걸리나
  • 태스크당 평균 LLM 호출: 25회 (플래너 6.4회 + 브라우저 에이전트 18.6회)
  • 성공 태스크 평균 소요 시간: 150초
  • 실패 태스크 평균 소요 시간: 220초
  • 자기 인식 실패(에이전트가 스스로 실패 보고): 52% 이상
실패 태스크가 성공보다 70초 더 걸린다는 건 에이전트가 실패를 즉각 인식하지 못하고 여러 번 시도한다는 의미다. 자기 인식 실패율 52%는 처음으로 웹 에이전트 평가에서 측정된 지표다.

네 가지 설계 원칙 정리

논문이 Agent-E 개발 경험에서 도출한 보편 원칙들이다. 웹 에이전트에만 적용되는 게 아니라 에이전틱 시스템 일반에 대한 주장이다.

원칙 1 — 도메인 특화 기본 스킬 (Domain-Specific Primitive Skills)

범용 도구를 여러 개 두는 것보다, 해당 도메인의 원자적 행동을 정확히 커버하는 특화 스킬을 만들어라. 웹에서는 클릭·입력·스크롤이 원자적 단위다. 로봇공학에서는 이동·파지·방향 전환이 원자적 단위가 된다.

단순해 보이지만, 스킬의 경계를 어디에 그을지가 실제로는 어려운 결정이다. 너무 세밀하게 쪼개면 LLM이 매번 여러 스킬을 조합해야 해서 오류가 늘어난다. 너무 추상화하면 예외 상황에서 실패한다.

원칙 2 — 계층적 아키텍처 (Hierarchical Architecture)

복잡한 태스크는 여러 추상화 레벨로 분리하라. 단, 복잡성이 정당화될 때만 계층을 추가하라. 단순 태스크에 계층 구조를 넣으면 오히려 오버헤드만 늘어난다.

병렬 실행 가능한 서브태스크 식별, 컴포넌트 격리, 에러 감지·복구가 계층화의 주요 이점이다.

원칙 3 — 페이로드 디노이징 (Payload Denoising)

LLM에게 전달하는 정보는 태스크에 맞게 필터링하라. 관련 없는 정보를 제거하고, LLM이 소비하기 좋은 형태로 변환하라. 이는 정확도와 비용 효율 두 가지에 모두 영향을 미친다.

DOM 디노이징은 이 원칙의 웹 에이전트 버전이다. 로봇공학이라면 센서 데이터 필터링이 될 것이고, 코드 에이전트라면 관련 파일만 컨텍스트에 포함하는 것이 된다.

원칙 4 — 언어적 행동 피드백 (Linguistic Action Feedback)

결과가 중요한 행동을 수행할 때마다 언어로 된 결과 보고를 제공하라. 이 피드백이 에이전트의 인과 추론을 직접적으로 도와준다.

원칙 적용 예시 — 웹 바깥의 에이전틱 시스템
도메인 원칙 2 (계층) 원칙 3 (디노이징) 원칙 4 (피드백)
로봇공학 계획↔실행 분리 관련 센서만 활성화 힘/위치 피드백
코드 에이전트 설계↔구현 분리 관련 파일만 컨텍스트 포함 테스트 결과 언어 보고
데스크톱 자동화 태스크↔스텝 분리 현재 앱 UI만 추출 창 상태 변화 보고

자기 개선 메커니즘

논문은 다섯 번째 영역으로 에이전트의 자기 개선(Self-Improvement)도 다룬다. 실패한 태스크 이력을 분석해 반복적인 실수 패턴을 찾고, 이를 프롬프트나 내부 규칙에 반영해 성능을 점진적으로 높이는 방향이다.

현재 구현은 완전히 자동화된 자기 개선보다는 로그 분석 기반의 수동 개선에 가깝다. 하지만 이 방향성 자체가 중요하다 — 에이전트가 자신의 실패를 인식하고(52% 이상이 자기 인식 실패로 보고됨), 그것을 학습에 활용하는 루프는 장기적으로 가장 강력한 개선 경로다.

· · ·

WebVoyager 이후의 맥락에서

이 논문은 2024년 7월에 나왔다. 우리가 이미 다룬 WebVoyager(57.1%, 멀티모달, GPT-4V)보다 16%p 높은 성능을 텍스트만으로 달성했다는 게 핵심이다.

이게 시사하는 바:

멀티모달이 항상 답이 아니다. 시각 정보를 쓰면 "보이는" 것에 의존하게 된다. 반면 DOM을 직접 읽으면 버튼의 실제 aria-label을 보고, 숨겨진 요소를 발견하고, 페이지 렌더링에 상관없이 구조를 파악한다. 웹이라는 도메인에서는 DOM이 시각보다 더 정확한 "진실"이다.

범용 에이전트가 특화 에이전트를 이겼다. WILBUR은 사이트별 특화 프롬프트로 52.6%였다. Agent-E는 범용 설계로 73.2%를 찍었다. 도메인 특화의 한계가 있고, 잘 설계된 범용 아키텍처가 그 한계를 넘어선다.

물론 Booking.com 27.3%가 보여주듯, 아직 갈 길이 멀다. 동적이고 복잡한 UI, 로그인이 필요한 태스크, 여러 탭을 오가는 플로우 등은 여전히 어렵다.

"There is likely room for further improvement in Agent-E using some of these strategies."

논문 자체가 인정한다. 멀티모달 통합, 강화학습, 더 나은 자기 개선 루프를 더하면 어디까지 갈 수 있을지 — 그게 이후 연구들이 탐구하는 방향이다.

저자: Aditya Vempaty, Scott Zimmerman, Oliver Rodriguez, Bianca Strohmer, Alex Grushin, Sailik Sengupta, Chitta Baral (Arizona State University, Stitch Fix)
