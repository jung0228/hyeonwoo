웹 에이전트 벤치마크 해부: LLM 출력이 점수가 되는 방법

Mind2Web의 네 가지 지표 수식, WebArena의 기능적 검증 로직, WebVoyager의 GPT-4V 심판 문제, 그리고 2025년 "Illusion of Progress" 논문이 폭로한 것들 — 벤치마크 숫자 뒤에서 실제로 무슨 일이 일어나는지 전부 분해한다.

Agent · Mar 7, 2026 · 32 min read

같은 "60%"가 다른 이유

논문을 읽다 보면 "Task Success Rate 60%"라는 숫자가 나온다. 그런데 어느 벤치마크의 60%인지에 따라 그 의미는 완전히 달라진다. Mind2Web에서 60%와 WebArena에서 60%는 측정하는 것 자체가 다르다. 그리고 WebVoyager에서 60%는 위 두 개와 또 다르다.

이 포스트는 그 차이의 실체를 파고든다. 평가 스크립트가 실제로 무엇을 확인하는지, LLM의 출력 하나가 어떤 계산 과정을 거쳐 점수가 되는지를 각 벤치마크마다 분해한다. 기본 개념(입출력 포맷, Playwright 통신, Mind2Web vs WebArena 파이프라인 개요)은 이전 포스트에서 다뤘으니, 여기선 평가 메커니즘 자체에 집중한다.

2023–2025년 주요 웹 에이전트 벤치마크를 평가 방식 기준으로 분류. Offline → Self-hosted → Live 순으로 "실제"에 가까워진다.

1. Mind2Web: 오프라인 평가의 수식 해부

Mind2Web(NeurIPS 2023)은 "웹 에이전트에게 실제 사람이 수행한 탐색 기록을 주고, 그 기록을 얼마나 잘 재현하는가"를 측정한다. 137개 사이트, 2,350개 태스크, 태스크당 평균 7.3개 스텝이 저장된 HTML 스냅샷으로 존재한다. 실제 브라우저는 없다.

세 개의 테스트 분할 — 일반화 난이도 계층

Mind2Web의 설계에서 종종 간과되는 부분이 테스트 분할이다. 단순히 train/test를 나누는 게 아니라 세 가지 일반화 시나리오로 테스트한다.

분할 조건 의미 태스크 수
Cross-Task 같은 사이트, 다른 태스크 유형 Amazon에서 훈련했는데 Amazon에서 본 적 없는 태스크 252
Cross-Website 같은 도메인, 다른 사이트 쇼핑몰 훈련 → 다른 쇼핑몰 테스트 177
Cross-Domain 전혀 다른 도메인 여행 사이트 훈련 → 금융 사이트 테스트 912

성능은 Cross-Task > Cross-Website > Cross-Domain 순으로 떨어진다. 논문에서 "우리 모델이 X%"라고 보고할 때 어느 분할인지를 반드시 확인해야 한다. Cross-Task 성능이 Cross-Domain보다 10–15%p 높은 경우가 흔하다.

네 가지 지표의 수식

Mind2Web은 스텝 하나를 (요소 선택, 액션 타입, 액션 값) 세 부분으로 분해하고, 각 레벨에서 정확도를 측정한다.

Element Accuracy (Ele. Acc)
Ele. Acc = 1 if predicted_element ∈ pos_candidates 0 otherwise
# pos_candidates: 같은 액션을 수행하는 의미론적으로 동등한 DOM 요소들의 집합 # 예: <label for="departure">와 <input id="departure">는 둘 다 정답
Operation F1 (Op. F1)
# CLICK: 단순 exact match (값이 없음) Op. F1(CLICK) = 1 if predicted_op == "CLICK" else 0 # TYPE / SELECT: SQuAD 스타일 토큰 레벨 F1 pred_tokens = tokenize(predicted_value) gt_tokens = tokenize(ground_truth_value) common = |pred_tokens ∩ gt_tokens| Precision = common / |pred_tokens| Recall = common / |gt_tokens| Op. F1 = 2 × Precision × Recall / (Precision + Recall)
Step Success Rate (Step SR)
Step SR = 1 if (Ele. Acc == 1) AND (Op. F1 == 1) 0 otherwise # 부분 점수 없음: 요소를 맞혀도 값이 틀리면 0
Task Success Rate (Task SR)
Task SR = 1 if Step SR == 1 for ALL steps in task 0 otherwise # 한 스텝이라도 틀리면 태스크 전체가 0

Task SR 폭락의 수학 — 왜 항상 낮은가

Step SR 50%라는 숫자만 보면 나쁘지 않아 보인다. 하지만 태스크당 평균 7.3개 스텝을 가정하면:

Task SR = (Step SR)^N. 스텝당 90% 정확도도 7스텝 태스크에서 Task SR은 48%로 떨어진다. 70%면 8%.

이 수학적 현실 때문에 Mind2Web 논문들은 대부분 Task SR보다 Step SR을 앞세운다. Task SR 5–10%는 "형편없다"는 뜻이 아니라 "스텝당 정확도 50–60%의 정상적인 귀결"이다. 두 숫자를 같이 봐야 한다.

MindAct 2단계 파이프라인이 왜 필요한가

페이지당 평균 1,135개의 DOM 요소를 LLM에 그대로 넣으면 컨텍스트 한계를 초과한다. MindAct(Mind2Web 논문의 베이스라인 시스템)는 이 문제를 2단계로 해결한다.

Stage 1 — Candidate Ranker (DeBERTa)
# 입력: 태스크 + 액션 이력 + DOM 요소 하나 # 출력: 관련성 스코어 (0~1) cross_encoder_input = f""" Task: {task_desc} Previous: {action_history} Element: {element_html_snippet} """ score = deberta_cross_encoder(cross_encoder_input) # 모든 요소에 대해 반복 → top-k 선택 # GPT-4: k=10, 소형 LM: k=50 candidates = sorted(elements, key=score)[:k]
Stage 2 — Action Predictor (GPT-4)
# 입력: top-k 후보를 객관식으로 구성 # 출력: (선택지 문자, 액션 타입, 값) prompt = f""" Task: Search one-way flights NYC to LAX Previous: [none] Choose the element: A. None of the above B. <div role="radio">One way</div> C. <input placeholder="Where from?"> D. <button>Search</button> E. <select name="stops">...</select> Action type: CLICK / TYPE / SELECT Value (if TYPE/SELECT): ___ """ output = gpt4(prompt) # → "B, CLICK, " (값 없음) # 평가: B == pos_candidates에 있나? # 있으면 Ele. Acc = 1
핵심 한계: 오프라인 평가는 실제 동작을 측정하지 않는다. 에이전트가 step 3에서 틀린 요소를 클릭했다면 실제 브라우저에서는 step 4의 페이지 상태가 완전히 달라진다. Mind2Web은 이걸 무시한다 — 각 스텝을 독립적으로 평가한다. 즉, "오류 복구 능력"이나 "이전 액션이 현재 페이지에 미치는 영향"은 전혀 측정되지 않는다.

실제 성능 표 (Cross-Website 기준)

모델 Ele. Acc Op. F1 Step SR Task SR
Flan-T5-XL (MindAct) 42.0% — 38.9% 5.1%
GPT-3.5-Turbo (MindAct) 47.3% — 42.8% 7.3%
GPT-4 (MindAct) 54.6% — 48.8% 9.7%
GPT-4 + SoM (MindAct) 58.1% — 52.3% 11.2%

Task SR가 Step SR에 비해 얼마나 낮은지 주목. Task SR 11%는 단순히 "나쁜" 게 아니라 (~52%)^7.3의 정상적인 결과다.

2. WebArena: 기능적 검증의 전체 구조

WebArena(ICLR 2024)의 근본적 혁신은 "에이전트가 무엇을 클릭했는가"가 아니라 "환경이 올바른 상태에 도달했는가"를 묻는다는 점이다. 이미 이전 포스트에서 파이프라인을 개괄했으니, 여기선 평가 메커니즘 내부를 더 깊이 들어간다.

전체 액션 스페이스

카테고리 액션 인자 비고
페이지 조작 click [elem] 요소 ID 또는 좌표 AX tree node ID 기준
type [elem] [text] 요소 ID + 입력 텍스트 기존 내용 지우고 입력
hover [elem] 요소 ID 드롭다운 열기 등
press [key] 키 조합 "Enter", "Ctrl+A" 등
scroll [dir] "up" / "down" 500px씩
탭 관리 new_tab 없음 새 탭 열기
tab_focus [n] 탭 인덱스 탭 전환
tab_close 없음 현재 탭 닫기
내비게이션 goto [url] URL 문자열 직접 URL 이동
go_back 없음 브라우저 뒤로
go_forward 없음 브라우저 앞으로
noop 없음 대기/관찰만
종료 stop [answer] 텍스트 답변 (선택) 태스크 완료 선언

두 가지 평가 함수 — r_info와 r_prog

WebArena 태스크의 56%는 정보 검색이 아닌 상태 변경(이슈 생성, 댓글 작성, 장바구니 조작 등)을 요구한다. 이 경우 텍스트 비교는 의미가 없고, 실제 환경 상태를 검사해야 한다. 논문은 두 함수를 정의한다.

r_info — 정보 검색 태스크 (44%)
# stop [answer]의 텍스트를 검증 # 세 가지 비교 방법 중 하나를 사용 ## 1. exact_match task: "What is the title of the most recent GitLab issue in proj A?" gt: "Fix login timeout bug" pred: "Fix login timeout bug" → r_info = 1.0 ## 2. must_include task: "List all open issues in proj A" gt: ["Fix login", "Add tests"] pred: "The open issues are: Fix login, Add tests, Update docs" → r_info = 1.0 # 포함하면 OK ## 3. fuzzy_match (GPT-4 심판) task: "Where is the nearest hospital?" gt: "UPMC Mercy" pred: "Mercy Hospital (UPMC)" → GPT-4 판정: 동일 → r_info = 1.0 # 표면형이 달라도 의미가 같으면 정답 # 논문: 인간 대비 99.7% 일치
r_prog — 상태 변경 태스크 (56%)
# 에이전트 실행 후 환경 상태를 직접 검증 # 네 가지 검증 방법 ## 1. GitLab API 호출 task: "Create issue 'Fix login bug' in project 'awesome-os'" validator: gitlab_api.get_issues( project="awesome-os" ).filter(title="Fix login bug") → 존재하면 1.0, 없으면 0.0 ## 2. DOM/JavaScript 검사 task: "Add Waffle Beanie to cart" validator: page.evaluate( "document.querySelector( '.cart-count').textContent" ) == "1" ## 3. URL 상태 확인 task: "Navigate to product #1234" validator: page.url.contains("/1234") ## 4. 데이터베이스 직접 쿼리 task: "Post comment on r/LocalLLaMA" validator: db.query( "SELECT * FROM comments WHERE user=? ORDER BY id DESC", [agent_user] ).first().text == expected_text
r_prog의 의의. 에이전트가 GitLab 이슈를 API로 직접 만들었든, 웹 폼을 채워 제출했든, 단축키를 썼든 — 결과물이 DB에 존재하면 정답이다. Mind2Web에서 "정답 경로"만 인정하던 것과 근본적으로 다르다. WebArena는 경로가 아닌 결과를 본다.

stop 액션과 불가능한 태스크 처리

WebArena에는 실제로 수행 불가능한 태스크가 포함된다. 예를 들어 존재하지 않는 사용자의 프로필을 찾으라는 요청. 이때 에이전트는 stop [N/A]을 출력해야 한다.

에이전트가 강제 종료되는 조건도 정의되어 있다: 동일 관찰에서 같은 액션을 3번 반복하거나, 파싱 불가능한 액션을 3번 연속 생성하면 태스크 실패 처리된다. 이 규칙은 무한 루프와 헛소리 생성을 방지한다.

불가능 태스크 힌트 제공 여부가 결과를 바꾼다. WebArena 논문의 ablation에서, 에이전트에게 "일부 태스크는 수행 불가능할 수 있다"는 힌트를 주면 성공률이 올라가지만, 이로 인해 실제로 가능한 태스크에서 에이전트가 너무 일찍 N/A를 선언하는 false negative가 증가한다. 각 논문에서 이 힌트를 줬는지 확인해야 한다.

3. WebVoyager: GPT-4V 심판의 문제

WebVoyager(ACL 2024)는 Mind2Web도, WebArena도 커버하지 못하는 지점을 겨냥했다: 실제 라이브 웹사이트에서 멀티모달 에이전트를 평가하는 것. Google, Amazon, GitHub, Booking.com 등 15개 실제 사이트, 643개 태스크. 에이전트는 스크린샷을 직접 보고 행동한다.

라이브 웹의 평가 문제 — 왜 GPT-4V가 심판이 됐나

WebArena는 Docker로 격리된 환경에서 DB 상태를 직접 검증한다. 하지만 실제 Amazon이나 GitHub에서는 그게 불가능하다. 페이지가 바뀌고, 로그인이 필요하고, 정답이 고정되지 않는다. WebVoyager는 이 문제를 GPT-4V를 심판으로 써서 해결했다.

GPT-4V 심판 프로세스:

GPT-4V 심판 입력
# 심판에게 주어지는 정보 [Task] Search for a round-trip flight from NYC to Paris, departing Dec 15, returning Dec 22. What is the cheapest option? [Agent's Final Answer] "The cheapest round-trip is $487 on Air France, departing at 10:30 AM." [Screenshots] [Image 1: Google Flights results page] [Image 2: Price filter applied] [Image 3: Final search result page] ... (마지막 k=전체 스크린샷) [Evaluation prompt] "Based on the task, the agent's answer, and the screenshots, did the agent successfully complete the task? Answer: Yes / No, with brief reason."
GPT-4V 심판 출력
# 성공 케이스 Yes. The agent successfully completed the task. Screenshot 3 shows Google Flights search results for NYC→Paris Dec 15–22. The agent's answer of $487 on Air France is visible in the results list and appears to be the lowest price. # 실패 케이스 예시 No. The agent navigated to the correct page but reported "$487" while the screenshot shows $512 as the actual cheapest option. The agent may have misread the price filter. # 인간 평가자와의 일치율: 85.3% # Cohen's kappa (전체 궤적): 0.70 # Cohen's kappa (마지막 1장만): 0.51

스크린샷 수 k가 평가 신뢰도를 결정한다

WebVoyager 논문에서 가장 중요한 발견 중 하나: GPT-4V에 몇 장의 스크린샷을 줄 것인가가 평가 신뢰도를 크게 좌우한다.

제공 스크린샷 인간과의 일치율 Cohen's kappa 실용성
마지막 1장만 ~73% 0.51 비용 저렴, 신뢰도 낮음
마지막 3장 ~80% 0.63 균형점
전체 궤적 85.3% 0.70 비용 높음, 가장 신뢰

kappa 0.51은 "moderate agreement"로, 동전 뒤집기보다 약간 낫다는 뜻이다. 마지막 스크린샷 하나만으로 판단하는 것은 신뢰하기 어렵다. 많은 후속 연구들이 비용 절감을 위해 마지막 1–3장만 쓰는데, 이 경우 평가 신뢰도는 크게 낮아진다.

Google Flights 70.7% 41.5% +29.2%p

Booking.com에서 텍스트 전용 에이전트가 2.3%라는 점이 눈에 띈다. Booking.com의 달력 UI와 가격 비교 레이아웃은 DOM만으로는 파악하기 매우 어렵다. 이런 시각적 의존도 높은 사이트에서 멀티모달의 이점이 극대화된다.

WebVoyager 결과에서 한 가지를 배운다면: 멀티모달의 이점은 균일하지 않다. 텍스트 밀도가 높은 사이트(ArXiv, GitHub)에서는 차이가 미미하고, 시각적 레이아웃에 의존하는 사이트(Booking.com, Google Flights)에서 극적인 차이가 난다.

4. VisualWebArena: 시각 이해가 필수일 때

WebArena 태스크들은 이론적으로 DOM/AX Tree만으로 전부 해결 가능하다. 이미지를 봐야 알 수 있는 정보가 없다. VisualWebArena(ACL 2024, Koh et al.)는 이 한계를 직접 공략한다: 시각 이해 없이는 원천적으로 풀 수 없는 태스크를 만든다.

이미지가 태스크 자체에 포함된다

VisualWebArena 태스크의 25.2%는 이미지를 태스크 설명에 직접 포함한다. 예를 들어:

태스크 예시: [이미지: 파란 스트라이프 셔츠 사진] "이 셔츠와 동일한 스타일인데 빨간색인 제품을 찾아서 장바구니에 담아라."

이 태스크는 AX Tree나 HTML에서 "파란 스트라이프 셔츠"라는 정보를 찾을 수 없다. 에이전트는 이미지를 보고, 그 시각적 특성(색상, 패턴)을 이해해야 한다.

새로운 평가 검증자

시각적 정답 검증을 위해 WebArena의 프로그래밍 검증자를 확장했다.

검증자 설명 사용 예
eval_vqa VQA 모델로 스크린샷에 질문 "이 상품의 색상은 빨간색인가?"
eval_fuzzy_image_match SSIM으로 스크린샷과 참조 이미지 비교 목표 제품 이미지와 현재 페이지 비교
must_exclude 특정 내용이 없어야 함 삭제된 포스트가 더 이상 안 보여야 함

결과 — 여전히 낮은 천장

에이전트 Task SR 비고
인간 88.70% 상한선
GPT-4o + SoM (최고) 19.78% 현재 SOTA
GPT-4V + SoM 16.37%
Llama-3-70B + SoM 9.78% 오픈소스 최고
Caption-augmented GPT-4 12.75% 이미지→텍스트 변환 후 텍스트 에이전트

인간 88.70% 대비 최고 모델 19.78%는 WebArena(78% vs 14%)보다 격차가 더 크다. Caption 방식(이미지를 텍스트로 변환한 뒤 텍스트 에이전트에 주기)이 12.75%에 그치는 것은, 이미지를 텍스트 설명으로 요약하면 시각적 세부 정보가 손실된다는 것을 보여준다.

5. WorkArena, WebLINX, WebCanvas — 각자 다른 문제

WorkArena (ICML 2024): 기업용 소프트웨어

WorkArena는 Fortune 500의 85%가 사용하는 ServiceNow 플랫폼에서 에이전트를 평가한다. 소비자 웹이 아닌 기업용 SaaS가 타겟이다. 33개 태스크 유형, 19,912개 태스크 인스턴스.

가장 충격적인 결과는 List-Filter 태스크에서 GPT-4o가 0%를 기록한다는 점이다. 드롭다운 5개 조건을 조합해 결과를 필터링하는 태스크인데, 에이전트들이 GUI 조작의 순서와 조합을 체계적으로 실행하지 못한다. 의미론적 이해는 있어도 "3번 이상의 필터를 정확한 순서로 적용한다"는 절차적 작업은 여전히 취약하다.

WorkArena는 BrowserGym이라는 범용 벤치마크 인프라도 함께 제공한다. 요소 ID 기반 액션, 좌표 기반 액션, Python/Playwright API를 통합 지원하며, 여러 벤치마크를 같은 코드베이스에서 실행할 수 있게 해준다.

WebLINX (ICML 2024 Spotlight): 대화형 웹 탐색

WebLINX는 혼자 태스크를 수행하는 에이전트가 아닌, 인간 사용자와 대화하면서 탐색하는 에이전트를 다룬다. 2,337개 전문가 데모, 155개 실제 사이트, 대화당 평균 43턴, 10만+ 인터랙션.

핵심 차별점: 에이전트가 say(text) 액션으로 사용자에게 말을 걸 수 있다. "지금 항공편 날짜를 확인하고 있어요. 편도로 드릴까요, 왕복으로 드릴까요?" 같은 질문이 가능하다.

평가는 턴 단위 미시 F1으로 측정한다: 액션 유형(Intent Match), 클릭 위치(Element IoU), 텍스트 값(Token F1)을 분해해서 합산한다. 미세 조정된 Llama-3-8B가 GPT-4V 제로샷(10.5%)을 크게 앞서는 28.8%를 달성한다는 점에서, 특정 도메인에서의 파인튜닝이 여전히 강력하다는 것을 보여준다.

WebCanvas / Mind2Web-Live (2024): 부분 점수 도입

WebArena의 이진 평가(성공/실패)는 중간에 7개 스텝 중 6개를 맞혀도 0점이다. WebCanvas는 이를 해결하기 위해 Key Node 개념을 도입한다: 태스크 완료에 필수적인 중간 상태들을 정의하고, 각 상태를 별도로 검증한다.

검증 방식 설명
ExactMatch URL, 선택된 요소 경로, 값이 정확히 일치
IncludeMatch 페이지 내용에 필요한 텍스트가 포함됨
SemanticMatch 임베딩 유사도로 의미론적 일치 확인

Mind2Web-Live는 Mind2Web 데이터셋을 라이브 웹사이트에서 재주석한 버전이다. 542개 태스크, 2,439개 Key Node. 그런데 재주석 과정에서 원본 태스크의 47% 이상이 사이트 변경이나 링크 깨짐으로 무효화됐다. 이건 정적 벤치마크의 수명이 얼마나 짧은지를 단적으로 보여준다.

Mind2Web-Live 최고 성능: Task SR 23.1%, Task Completion Rate(Key Node 기반 부분 점수) 48.8%. Task SR이 낮아도 Completion Rate는 훨씬 높다 — 에이전트가 태스크를 "절반 이상" 완료하고 있다는 의미다.

6. "Illusion of Progress" — 2025년의 메타 비판

2025년 4월, OSU NLP 그룹은 "An Illusion of Progress? Assessing the Current State of Web Agents"(COLM 2025)를 발표한다. 논문의 주장은 단순하지만 충격적이다: 현재 보고된 웹 에이전트 성능 수치의 상당 부분이 허상이다.

WebVoyager의 구조적 문제

논문은 WebVoyager를 체계적으로 해부한다. 15개 사이트, GPT-4V 심판 — 이 구조에서 무엇이 잘못됐는가?

핵심 발견: Google Search만 쓰는 에이전트가 WebVoyager에서 51% 달성. "Google에서 검색하고 첫 결과 페이지의 정보를 답변으로 제출하는" 단순한 에이전트가 WebVoyager 태스크의 절반 이상을 통과한다. 즉 WebVoyager 태스크의 상당 부분은 실제 멀티스텝 웹 탐색이 필요 없다. 검색 한 번으로 해결된다.

이 발견은 "우리 에이전트가 WebVoyager에서 X%를 달성했다"는 수많은 논문 클레임을 다시 봐야 한다는 의미다. 그 중 얼마가 Google Search 능력이고, 얼마가 진짜 웹 탐색 능력인지 알 수 없다.

Online-Mind2Web — 더 엄격한 기준

논문은 WebVoyager의 대안으로 Online-Mind2Web 벤치마크를 제안한다. 설계 원칙:

  • 300개 태스크, 136개 사이트 (다양성 확보)
  • Mind2Web 원본에서 필터링(47% 무효화 후 유효한 것만 사용)
  • 모든 태스크를 라이브 웹사이트에서 인간이 직접 검증
  • 난이도 분류: 쉬움(≤5스텝) 83개, 중간(6–10스텝) 143개, 어려움(≥11스텝) 74개

WebJudge — 더 신뢰할 수 있는 자동 평가

기존 자동 평가자들(GPT-4V judge 등)은 인간 판단과의 일치율이 66–79%였다. WebJudge는 이를 개선한 새 방법론이다.

WebJudge 프로세스
# 1단계: 요구사항 추출 task = "Find a Python tutorial for beginners on YouTube, sort by view count, find the top video" requirements = o4_mini.extract(task) → ["navigate to youtube.com", "search 'python tutorial beginners'", "apply sort by view count filter", "identify top result title & views"] # 2단계: 관련 스크린샷 필터링 for screenshot in trajectory: relevance = score(screenshot, req) if relevance > δ: # δ=3 keep(screenshot) # 3단계: o4-mini 이진 판정 verdict = o4_mini.judge( task=task, requirements=requirements, key_screenshots=filtered_screenshots, agent_answer=final_answer ) → {"success": True/False, "reason": "..."}
평가자 비교
# 인간 판단과의 일치율 기존 GPT-4V judge: 66~79% WebJudge (o4-mini): 86% WebJudge-7B (RM): 83% # 평균 성공률 오차 (인간 대비) 기존 방법: ±5~12%p WebJudge: ±3.8%p # 왜 나아지는가? - 관련 스크린샷만 선택 (노이즈 제거) - 요구사항을 명시적으로 추출해 판단 기준을 구체화 - o4-mini의 더 나은 멀티모달 추론 # WebJudge-7B는 RL 훈련용으로도 사용 가능 # (오픈소스 reward model)

진짜 숫자들 — 제대로 측정하면 얼마나 나오나

에이전트 발표된 수치 Online-Mind2Web
(인간 평가)		 차이
OpenAI Operator WebArena ~65% 61.3% 가장 높음
Claude CU 3.7 WebArena ~58% 56.3% 비교적 일관
Browser Use 자체 벤치 ~80% ~30% 큰 격차

"Illusion of Progress"의 핵심 주장. 2024년에 나온 수많은 웹 에이전트들이 엄격한 평가에서 2024년 초 베이스라인인 SeeAct와 비슷한 성능을 보인다. 발표된 높은 수치들의 상당 부분은 쉬운 벤치마크(WebVoyager 등)나 느슨한 자동 평가의 인플레이션이다. 실제 라이브 웹에서 인간이 검증하면 수치가 크게 내려온다.

7. 평가의 근본적 긴장들

오프라인 vs 온라인 — 수치로 본 격차

Mind2Web-Live 논문에서 측정한 결과: 동일한 모델이 오프라인(캐시된 HTML) 평가와 라이브 웹사이트 평가에서 얼마나 다른 점수를 보이는지를 직접 비교했다.

모델 오프라인 Step SR 라이브 Step SR 하락폭
GPT-4 (MindAct) 48.8% 25.5% -23.3%p
GPT-3.5 (MindAct) 42.8% 19.1% -23.7%p

오프라인 성능의 절반 수준으로 떨어진다. 원인은 복합적이다: 사이트 UI 변경, 동적으로 생성되는 요소 ID, 팝업과 쿠키 배너, 로그인 요구, 그리고 오프라인 평가에서는 없던 "이전 액션의 영향으로 변한 페이지 상태". 오프라인 벤치마크 숫자는 실제 라이브 성능의 상한선이 아니라 과대 추정치라고 봐야 한다.

참조 기반 vs 기능적 평가

두 평가 방식은 근본적으로 다른 것을 측정한다.

측면 참조 기반 (Mind2Web) 기능적 (WebArena)
측정 대상 인간 주석자가 택한 경로와의 유사도 환경이 목표 상태에 도달했는가
대체 경로 인정 안 함 (정답 경로 하나만) 완전히 수용 (어떤 경로든 OK)
패널티 오류 다른 경로로 성공해도 실패 경로 무관, 결과만 본다
부분 점수 스텝 단위 (Element Acc, Step SR) 없음 (이진 성공/실패)
재현성 높음 (고정된 HTML 스냅샷) 낮음 (환경 상태 오염 가능)
인프라 HuggingFace 데이터셋 다운로드만 Docker, 포트 설정, 환경 초기화 필요

평가 오염과 포화의 문제

2024–2025년 사이 여러 벤치마크가 포화에 가까워지거나 이미 포화됐다. WebVoyager의 경우 90%+ 성능을 클레임하는 에이전트들이 등장했다 — 그런데 Google Search 전용 에이전트가 51%라면, 남은 40%p도 그다지 신뢰하기 어렵다.

오염(contamination) 문제도 실재한다. Mind2Web의 훈련 분할은 공개되어 있고, 사이트 이름과 태스크가 그대로 들어 있다. 대형 모델이 이 데이터를 사전훈련 중에 봤을 가능성을 완전히 배제할 수 없다. Mind2Web이 테스트 분할을 비밀번호로 보호하는 것도 이 때문이다 — 하지만 훈련 분할 자체의 오염은 막을 수 없다.

현재 연구 커뮤니티가 취하는 mitigation 전략들:

  • 동적 태스크 재생성: Online-Mind2Web은 오래된 태스크를 정기적으로 교체한다
  • 테스트 분할 비공개: 비밀번호 보호 또는 API를 통한 제출만 허용
  • 훈련 컷오프 이후 데이터: 모델 훈련 완료 후에 생성된 태스크 사용 (SWE-Bench-Live 방식)
  • 자체 호스팅 환경: 특정 시점의 데이터로 초기화된 Docker 환경

부분 점수의 딜레마

이진 Task SR(WebArena)은 7스텝 중 6스텝 완료도 0점이다. 너무 엄격하다. 하지만 부분 점수(WebCanvas의 Key Node)는 "어디까지가 필수 중간 상태인가"를 정의하는 문제를 만든다. Key Node를 어떻게 정의하느냐에 따라 같은 에이전트의 점수가 달라진다.

LLM 심판 방식(GPT-4V가 부분 완료 정도를 점수화)은 더 나쁜 문제를 가져온다: LLM 심판은 애매한 케이스에서 일관성 없이 중간 점수를 주는 경향이 있다. "절반 정도 완료했다"는 판단 자체가 주관적이고, 재현되지 않는다.

어떤 숫자를 믿어야 하는가

웹 에이전트 논문의 성능 수치를 읽을 때 체크해야 할 것들을 정리하면:

체크 항목 의미 위험 신호
어느 벤치마크? 오프라인/온라인, 라이브/격리 여부 WebVoyager 단독 클레임
어느 분할? Mind2Web: Cross-Task vs Cross-Domain 가장 쉬운 분할만 보고
어느 지표? Task SR vs Step SR (10배 차이 가능) Step SR만 보고
평가자는? 인간 vs GPT-4V vs 프로그래밍 검증 자체 LLM 심판, 단일 평가자
불가능 태스크? 힌트 제공 여부가 결과를 바꿈 명시 안 함
인간 상한선? 같은 조건에서의 인간 성능 인간 수치 없음

가장 신뢰할 수 있는 숫자는 Online-Mind2Web처럼 (a) 라이브 웹사이트, (b) 다양한 사이트, (c) 인간이 직접 검증한 것이다. 현재 그 기준에서 최고는 Operator의 61.3%이고, 이게 웹 에이전트의 진짜 현주소에 가장 가깝다. 인간의 78%와 아직 상당한 격차가 있다.

벤치마크는 항상 현실의 단순화다. 그 단순화가 얼마나 합리적인지, 그리고 측정값이 우리가 실제로 신경 쓰는 것과 얼마나 상관관계가 있는지를 — 숫자를 보기 전에 물어야 한다.
