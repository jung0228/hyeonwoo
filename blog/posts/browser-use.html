Browser Use: 오픈소스 웹 자동화 프레임워크 해부

4일 만에 만든 데모가 Hacker News를 뒤집었다. 3개월 뒤 GitHub stars 50,000. ETH Zurich 대학원생 두 명이 만든 프레임워크가 오픈소스 웹 에이전트의 사실상 표준이 된 과정과, 그 내부 구조를 분석한다.

Agent · LLM · Mar 2026 · 22 min read

4일 데모, 그리고 폭발

2024년 11월, Magnus Müller와 Gregor Žunič는 ETH Zurich 데이터 사이언스 석사 과정 중이었다. 두 사람은 4일 만에 데모를 만들어 Hacker News에 올렸다. 인터넷이 폭발했다.

3개월 뒤 GitHub stars 50,000개. 2025년 3월에는 중국 AI 에이전트 플랫폼 Manus AI가 Browser Use를 핵심 컴포넌트로 사용한다는 사실이 알려지면서 글로벌 인지도가 다시 한 번 급등했다. 같은 달 Felicis 주도로 $17M 시드 라운드가 클로즈됐다. Paul Graham도 참여했다.

현재 GitHub stars 80,000+. MIT 라이선스, 오픈소스. Fortune 500 기업들이 신뢰하여 사용한다고 밝히고 있다. Y Combinator 등록 스타트업.

핵심 사명 — "Make websites accessible for AI agents"
기존 자동화 도구는 경직된 스크립트에 의존한다. 사이트가 조금만 바뀌면 깨진다. Browser Use의 목표는 LLM이 사람처럼 웹을 탐색할 수 있게 하는 것 — 스크립트 없이, 규칙 없이.
· · ·

Agent-E와 무엇이 다른가

우리가 이미 분석한 Agent-E는 두 개의 LLM 에이전트가 계층을 이루는 구조였다. Planner Agent가 태스크를 분해하고 서브태스크를 위임하면, Browser Navigation Agent가 실제 브라우저를 조작한다. 역할이 명확히 분리된 계층적 아키텍처다.

Browser Use는 다르다. 단일 에이전트 루프. 하나의 Agent가 모든 결정과 실행을 담당한다. 생각하고, 액션을 결정하고, 실행하고, 결과를 기록한다. 플래너도 없고, 위임도 없다.

Agent-E Browser Use
아키텍처 계층적 (Planner + Browser Nav) 단일 에이전트 루프
설계 철학 역할 분리, 오류 격리 단순성과 유연성 우선
LLM 호출 스텝당 2회 (Planner + Navigator) 스텝당 1회
오류 복구 플래너 레벨 재계획 에러 주입 + Fallback LLM
확장성 구조화된 서브태스크 커스텀 액션 등록으로 무한 확장
WebVoyager 성능 73.2% (2024년 당시 SOTA) 89.1%

어떻게 단일 에이전트가 계층적 아키텍처를 이겼을까. 답은 아키텍처 선택보다 엔지니어링 디테일에 있다. DOM 처리 방식, 브라우저 통신 레이어, 루프 안정화 메커니즘 — 이것들이 성능을 결정한다.

5단계 실행 사이클

Browser Use의 에이전트 루프는 매 스텝마다 5개 단계를 반복한다.

Phase 1 — State Capture: BrowserSession.get_state()를 호출해 현재 DOM 상태를 수집한다. DomService가 DOM 트리를 추출하고, 필요한 경우 스크린샷을 캡처한다.

Phase 2 — Prompt Build: MessageManager가 LLM 컨텍스트를 조합한다. 시스템 프롬프트, 태스크 설명, 현재 DOM 표현(최대 200개 인터랙티브 요소), 실행 히스토리 요약, 사용 가능한 액션 목록이 담긴다.

Phase 3 — LLM Call: Pydantic 기반 AgentOutput 스키마로 structured output을 요청한다. LLM은 네 가지를 반환한다: thinking(내부 추론), evaluation_previous_goal(이전 액션 평가), memory(영속 메모리 노트), action(실행할 액션 목록).

Phase 4 — Action Execute: Tools 레지스트리가 각 액션을 EventBus를 통해 디스패치한다. BrowserSession이 CDP 명령으로 브라우저에 실행한다. 성공/실패 여부와 extracted_content를 반환한다.

Phase 5 — History Record: AgentHistory 레코드를 생성한다. LLM 출력, 액션 결과, 브라우저 상태 스냅샷, 타이밍 메타데이터가 기록된다. 다음 스텝의 Phase 2에서 히스토리 요약으로 사용된다.

· · ·

DOM 처리: 10,000+에서 200으로

Browser Use가 가장 공을 들인 부분이다. 웹페이지는 보통 10,000개 이상의 DOM 노드를 가진다. 이것을 그대로 LLM에게 넘기면 두 가지 문제가 생긴다 — 토큰 비용이 폭발하고, 관련 없는 정보로 인해 판단 정확도가 떨어진다.

Browser Use는 이것을 ~200개의 인터랙티브 요소로 압축한다. 토큰 사용량 기준 약 95% 절감이다.

이 압축이 중요한 이유가 있다. 기존 웹 에이전트들은 XPath나 CSS 셀렉터로 요소를 지정한다. LLM이 "#nav > ul > li:nth-child(3) > a" 같은 복잡한 셀렉터를 생성해야 하는데, 페이지가 조금만 바뀌면 깨진다. 그리고 LLM은 이런 셀렉터를 자주 잘못 생성한다(hallucination).

Browser Use는 숫자 인덱스로 대체한다. LLM이 "[2]를 클릭"이라고 하면 끝이다. 인덱스는 각 스텝마다 재할당되므로 영속성이 필요 없다. 단순하고, 오류가 줄어든다.

백엔드 식별자: backendNodeId
각 노드는 CDP의 backendNodeId를 통해 추적된다. DOM이 동적으로 변경되어도 노드 교체가 감지되면 즉시 재추출한다. 숫자 인덱스는 LLM을 위한 인터페이스고, 내부적으로는 CDP 영속 식별자가 정확도를 보장한다.
· · ·

Playwright를 버린 이유

Browser Use는 초기에 Playwright를 사용했다. 많은 웹 자동화 프로젝트처럼. 그런데 프로덕션에서 한계가 드러났다. 결국 직접 CDP(Chrome DevTools Protocol)와 통신하는 자체 라이브러리를 만들었다.

Playwright의 세 가지 문제

① 이중 홉 구조: Playwright의 Python 바인딩은 Python → Node.js WebSocket 서버 → CDP 순서로 이중 릴레이된다. 에이전트 루프에서 수천 번 CDP 호출이 발생하면 — element 위치 확인, opacity 체크, paint order 확인, JS event listener 조회, aria 속성 확인 — 지연이 누적된다.

② 엣지 케이스 실패: 16,000px를 넘는 스크린샷, dialog 처리, cross-origin iframe, 원격 브라우저에서의 파일 작업. Playwright 추상화 레이어 안에서는 해결할 수 없는 문제들이었다.

③ State drift: Python-Node.js-Chrome 간 이중 RPC 릴레이는 동기화 실패를 야기한다. 탭 크래시나 포커스 없이 시작하는 작업에서 Node.js 프로세스가 무한 대기 상태에 빠지는 경우가 있었다.

cdp-use: 자체 CDP 클라이언트

Browser Use 팀이 직접 cdp-use라는 type-safe Python CDP 클라이언트를 만들었다. 핵심 변화 두 가지:

이벤트 기반 Watchdog 패턴: 폴링 대신 CDP 이벤트를 구독한다. downloads_watchdog, crash_watchdog 등이 각각 전담 이벤트를 처리한다. 브라우저 상태 변화를 실시간으로 감지하면서도 CPU를 낭비하지 않는다.

Super-selector: 각 DOM 노드를 단일 식별자 대신 복합 식별자로 추적한다. targetId(탭), frameId(프레임), backendNodeId(노드), 좌표, 폴백 CSS 셀렉터가 모두 포함된다. 이 덕분에 Playwright가 처리하지 못하던 cross-origin iframe도 정확하게 조작할 수 있다.

"CDP direct communication means we control everything — no abstraction layer can break between us and the browser."

DOM 우선, Vision은 보조

Browser Use의 핵심 설계 철학: DOM이 먼저다, 스크린샷은 필요할 때만.

많은 웹 에이전트가 스크린샷을 기반으로 동작한다. 보기에는 직관적이다 — 사람이 웹을 보는 방식과 같으니까. 그런데 스크린샷 한 장이 LLM 파이프라인에 추가될 때마다 이미지 인코더 처리 시간이 붙는다. Browser Use 측정값으로 스텝당 약 0.8초.

OnlineMind2Web 벤치마크 기준 Browser Use의 평균 태스크 완료 시간은 68초다. Gemini Computer Use 225초, Claude Sonnet 4.5 285초, OpenAI Computer-Using Model 330초. 속도 차이의 상당 부분이 이 선택에서 온다.

언제 스크린샷을 쓰는가
  • CAPTCHA 해석
  • 광고 팝업이나 모달이 DOM 구조를 가릴 때
  • Canvas 기반 UI처럼 DOM에 의미있는 구조가 없을 때
  • use_vision="auto" 설정 시 에이전트가 스스로 판단

extract 도구도 같은 철학을 따른다. 페이지에서 특정 정보를 추출할 때 스크린샷을 찍지 않고, DOM에서 직접 markdown 쿼리를 실행한다. 별도의 LLM 호출로 관련 데이터만 뽑아낸다. 20,000+ 토큰짜리 페이지도 효율적으로 처리한다.

세 가지 엔지니어링 디테일

Browser Use의 성능을 실제로 만드는 것들이다. 겉으로는 보이지 않지만, 없으면 바로 무너진다.

ActionLoopDetector — 루프 탈출 장치

에이전트가 같은 액션을 반복하다가 빠져나오지 못하는 것은 웹 에이전트의 고질적 문제다. 같은 버튼을 계속 클릭한다. 같은 URL을 계속 방문한다. Browser Use는 액션 시퀀스와 페이지 fingerprint를 모니터링하다가 반복 패턴을 감지하면, 다음 LLM 프롬프트에 "nudge" 메시지를 주입한다. 에이전트가 다른 접근을 시도하도록 유도하는 것이다.

Message Compaction — 장기 태스크 토큰 관리

에이전트가 15 스텝을 실행하면 히스토리만으로 43,000+ 토큰이 된다. 이 상태가 계속되면 컨텍스트 한계에 도달하거나 비용이 폭발한다. Browser Use는 히스토리가 임계값을 넘으면 보조 LLM을 불러서 오래된 스텝들을 요약한다. 압축 후 토큰 사용량은 약 12,600으로 안정화된다. 장기 태스크에서 토큰 증가가 선형이 아니라 거의 상수 수준으로 유지된다.

KV Cache 인식 프롬프트 구조

LLM API의 KV Cache는 프롬프트 앞부분이 변하지 않아야 히트율이 높아진다. Browser Use는 이것을 의식적으로 설계에 반영했다. 히스토리(상대적으로 변하지 않는 부분)를 프롬프트 앞에 배치하고, 동적으로 바뀌는 DOM 상태를 뒤에 배치한다. 액션 출력 토큰도 최소화했다 — 액션 명세당 10~15 토큰만 사용한다. 입력 토큰(29.1ms/1K)보다 출력 토큰(62.6ms/10)이 215배 비싸기 때문이다.

성능: WebVoyager 89.1%

WebVoyager 벤치마크 586개 태스크 기준이다.

에이전트 성공률 타입
Browser Use (GPT-4o) 89.1% 단일 루프, 오픈소스
OpenAI Operator 87% 상용 서비스
Agent-E 73.2% 계층적, 논문
Anthropic Computer Use 56% 스크린샷 기반

사이트별로 편차가 있다. Huggingface(100%), Google Flights(95%), Amazon(92%), GitHub(92%)처럼 구조적이고 예측 가능한 사이트에서 높다. Booking.com(80%)처럼 동적 달력 UI, 복잡한 날짜 선택이 필요한 사이트에서 낮다.

2026년 초 기준 WebVoyager 리더보드 전체 1위는 Surfer 2(97.1%)다. Browser Use는 5위권이다. 오픈소스 프로젝트 기준으로는 사실상 최상위다.

OnlineMind2Web 벤치마크에서는 65.7% 정확도, 평균 68초로 완료했다. Computer Use 계열(225~330초)과 속도 차이가 크다. DOM 직접 파싱과 선택적 Vision의 결과가 여기서 나타난다.

커스텀 액션과 확장성

Browser Use의 기본 액션 스페이스는 단순하다.

  • navigate(url) — URL 탐색
  • click(index) — 숫자 인덱스로 요소 클릭
  • input_text(index, text) — 텍스트 입력
  • extract(goal) — 콘텐츠 추출
  • scroll(direction) — 스크롤
  • done(result) — 태스크 완료 선언

여기서 확장이 시작된다. @tools.action 데코레이터로 커스텀 액션을 등록하면, 에이전트가 일반 액션처럼 사용할 수 있다. 데이터베이스 저장, 외부 API 호출, 파일 시스템 접근 — 무엇이든 가능하다.

from browser_use import Agent, Browser, Tools
from langchain_openai import ChatOpenAI

tools = Tools()

@tools.action(description='검색 결과를 파일에 저장')
async def save_result(content: str) -> str:
    with open('results.txt', 'a') as f:
        f.write(content + '\n')
    return f"저장됨: {content[:50]}..."

# 퍼시스턴트 세션 — 여러 태스크에서 브라우저 재사용
browser = Browser(keep_alive=True)
llm = ChatOpenAI(model="gpt-4o")

agent = Agent(
    task="각 경쟁사 제품 가격 조사하여 저장",
    llm=llm,
    browser=browser,
    tools=tools,
    allowed_domains=["competitor1.com", "competitor2.com"],
)
result = await agent.run()
print(result.final_result())

이 구조가 단일 에이전트 루프의 한계를 상당 부분 보완한다. 복잡한 태스크를 플래너 없이 처리하려면, 커스텀 액션으로 "인라인 플래닝"을 구현하는 것이다. 결과를 중간에 파일에 저장하고, 다음 단계에서 그것을 읽어서 판단하는 식이다.

MCP 서버 통합

Browser Use는 Model Context Protocol(MCP) 서버도 제공한다. Claude, Cursor, Windsurf 같은 AI 개발 도구에서 Browser Use 자동화를 직접 호출할 수 있다.

# 로컬 셀프호스팅 MCP 서버 실행
uvx browser-use --mcp

MCP를 통해 노출되는 도구: browser_task(자동화 실행), list_browser_profiles(클라우드 프로파일 조회), monitor_task(실시간 태스크 상태). Claude Code 세션에서 Browser Use 태스크를 직접 실행할 수 있다는 의미다.

오픈소스 웹 에이전트의 지금

Browser Use가 흥미로운 이유는 성능만이 아니다. 이 프로젝트가 보여주는 것은 학술 논문과 오픈소스 엔지니어링의 방향 차이다.

Agent-E는 에이전틱 시스템 설계 원칙을 도출하는 것이 목표였다. 계층적 아키텍처, 페이로드 디노이징, 언어적 피드백 — 추상적이고 보편적인 원칙. Browser Use는 그 원칙들을 실제로 구현하면서, 동시에 그것들을 넘어서는 엔지니어링 결정을 내렸다.

Playwright 대신 CDP 직접 통신. DOM 압축 파이프라인. KV Cache 인식 프롬프트 구조. ActionLoopDetector. 이것들은 어느 논문에도 "원칙"으로 등장하지 않는다. 프로덕션 환경에서 실패를 반복하면서 나온 것들이다.

브라우저 자동화의 수요는 계속 커질 것이다. 현재 Browser Use의 한계는 여전히 동적이고 복잡한 UI, 장기간 인증이 필요한 플로우, 여러 탭을 동시에 다루는 병렬 작업이다. Manus AI 같은 상업적 플레이어들이 이 위에 무엇을 올리는지가 다음 챕터다.

한편, 단순성 자체가 전략이었다. 계층적 아키텍처보다 단일 루프가 더 많은 LLM을 지원하기 쉽고, 더 많은 커스텀 액션을 수용하고, 더 많은 사용자가 빠르게 시작할 수 있다. GitHub stars 80,000개가 그것을 증명한다.
