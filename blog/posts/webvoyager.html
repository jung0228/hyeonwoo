WebVoyager: 진짜 웹에서 동작하는 멀티모달 에이전트

GPT-4V로 스크린샷을 직접 보고, 실제 웹사이트를 탐색하다.

Agent · Multimodal · LLM · Mar 2026 · 18 min read

웹을 탐색하는 에이전트, 무엇이 어려운가

웹을 자동으로 탐색하는 에이전트를 만드는 건 생각보다 복잡하다. 사람은 웹페이지를 눈으로 보고 직관적으로 이해하지만, 컴퓨터에게 웹페이지는 수천 줄의 HTML 코드 덩어리다. "항공편 검색하고 최저가 찾아줘" 같은 태스크를 자동화하려면 에이전트가 버튼을 인식하고, 달력을 읽고, 검색 결과를 이해하고, 여러 페이지를 넘나들어야 한다.

2024년 발표된 WebVoyager (ACL 2024)는 이 문제에 멀티모달 LLM을 직접 투입한 첫 번째 end-to-end 시스템 중 하나다. 논문의 핵심 관찰은 이렇다:

"Rendering HTML into visual webpages is critical because rendered web pages are inherently designed with user experience (UX), emphasizing intuitive information and structured presentation."

HTML을 텍스트로 읽는 것과, 실제로 렌더링된 화면을 보는 것은 근본적으로 다르다는 이야기다.

· · ·

이전 방법들: 뭐가 문제였나

WebVoyager 이전에도 웹 에이전트 연구는 많았다. 2021년 WebGPT부터 2023년 Mind2Web까지.

시스템 연도 접근 방식 한계
WebGPT 2021 텍스트 기반 + fine-tuned GPT-3 시각 정보 없음, 검색에 특화
WebAgent 2023 T5로 HTML 압축 + Flan-U-PaLM 텍스트만, 파인튜닝 필요
WebGUM 2023 T5 + ViT (스크린샷 + HTML 함께) 시뮬레이션 환경, 파인튜닝 필요
PIX2ACT 2023 스크린샷 전용 MiniWoB++ 한정, 파인튜닝 필요
Mind2Web 2023 DeBERTa 필터 + Flan-T5 오프라인 스냅샷, 고정 경로 평가
WebVoyager 2024 GPT-4V + 실제 웹 + 스크린샷 파인튜닝 불필요, 실시간 웹

두 가지 문제가 반복됐다.

문제 1 — 에이전트가 보는 것과 사람이 보는 것이 달랐다

기존 에이전트들은 HTML/DOM 텍스트만 입력으로 받았다. 문제는:

"Existing approaches overlook a critical functionality of browsing: rendering HTML into visual webpages."

달력이 몇 월인지, 바 차트의 높이가 뭘 의미하는지, 어느 버튼이 실제로 클릭 가능한지 — 이런 정보들은 HTML 텍스트에서 읽어내기 매우 어렵거나 불가능하다.

문제 2 — 평가 자체가 현실과 달랐다

Mind2Web 같은 기존 벤치마크는 두 가지 구조적 문제를 안고 있었다:

"Existing benchmarks only focus on stepwise and offline evaluation...This approach may not fully account for the variety of viable strategies to accomplish a task."

오프라인 스냅샷 — 실시간 웹이 아니라 캐시된 정적 페이지에서 평가. 고정 경로 비교 — 정해진 정답 경로와 step 단위로 비교하기 때문에, 다른 경로로 같은 결과에 도달해도 실패로 처리된다.

WebVoyager 시스템 설계

Set-Label: 번호를 붙여 클릭하게 하다

스크린샷을 그냥 통째로 넣으면 모델이 "어디를 클릭해야 하지?"를 정확히 지시하기 어렵다. 좌표를 직접 출력하게 할 수도 있지만, 그러면 픽셀 정밀도 문제가 생긴다.

WebVoyager의 해결책은 Set-Label 방법이다. GPT-4V-ACT라는 JavaScript 기반 툴을 써서, 페이지에서 인터랙티브 요소(버튼, 링크, 입력창 등)를 자동으로 감지해 번호가 붙은 박스를 오버레이한다.

Set-Label이 각 요소에서 추출하는 정보
  • 숫자 레이블 — 1, 2, 3... (모델이 "Click [3]" 식으로 참조)
  • 텍스트 콘텐츠 — 요소 내 텍스트
  • 요소 타입 — button, link, input 등
  • aria-label 속성 — 접근성 레이블 (있는 경우)
박스 색깔은 검정색으로 고정. 실험해보니 여러 색 쓰는 것보다 단일 검정색이 성공률이 더 높았다고.

덕분에 모델 출력이 명확해진다:

Click [3]
Type [1]; New York to London
Scroll [WINDOW]; down

Object detection 없이 JS 룰 기반으로 추출하기 때문에 가볍고 빠르다.

액션 스페이스: 딱 7가지

7가지 액션

Click [N] — N번 요소 클릭 (링크, 버튼 등)

Type [N]; [텍스트] — N번 입력창 선택 후 기존 내용 지우고 입력, 자동으로 Enter까지 누름

Scroll [N 또는 WINDOW]; [up/down] — 특정 요소 or 전체 페이지 스크롤

Wait — 페이지 로딩 대기

GoBack — 이전 페이지로 이동

Google — 구글 검색으로 새로 시작 (막혔을 때 탈출용)

ANSWER; [내용] — 최종 답변 제출, 태스크 종료

Google 액션이 흥미롭다. 길을 잃었을 때 구글로 돌아가서 다시 시작하는 탈출구다. 사람도 웹에서 막히면 구글로 돌아가잖아.

ReAct 스타일 루프: 생각하고 행동하기

매 step마다 GPT-4V는 Thought (현재 상황 분석) → Action (실행할 액션 코드) 순서로 출력한다. 최대 15 step 까지 반복하고, 목표를 달성하면 ANSWER로 종료.

GPT-4V에 들어가는 입력 형식은 수식으로 표현하면:

ct = (o₁, a₁, ..., ot₋₁, at₋₁, ot, I)

  • oi = 각 step의 스크린샷 (observation)
  • ai = 이전 Thought + Action 텍스트
  • I = 유저의 태스크 지시문

Context Clipping: 15장을 다 넣을 순 없다

15 step을 돌면 스크린샷 15장이 쌓인다. 각 스크린샷이 1024×768px이고, 15 step 풀 트레젝토리를 처리하려면 7,000+ 토큰이 필요하다. GPT-4V의 context window에 쑤셔 넣으면 비용 폭탄이고, 오픈소스 모델(LLaVA의 4096 토큰 한계)로는 애초에 불가능하다.

해결책: Context Clipping — 스크린샷은 최근 3장만, Thought+Action 텍스트는 전부 유지.

왜 이게 합리적인가? "뭘 했는지" (행동 이력)는 전부 기억해야 루프를 안 돈다. 하지만 화면 스크린샷은 최근 것만 있어도 지금 어디 있는지 파악하는 데 충분하다. 오래된 스크린샷은 이미 지나간 페이지니까.

단, 이로 인한 부작용도 있다. 논문이 직접 인정하는 내용: "agent tends to repeat its previous mistakes due to the input clipping" — 너무 오래전에 했던 실수는 기억 못 해서 같은 실수를 반복하는 경우가 생긴다.

벤치마크: 어떻게 643개 태스크를 만들었나

WebVoyager는 자체 벤치마크를 함께 제안한다. 15개 실제 웹사이트, 643개 태스크. 로그인이나 CAPTCHA가 필요 없는 사이트들만 선택했다.

15개 사이트: Allrecipes, Amazon, Apple, ArXiv, BBC News, Booking, Cambridge Dictionary, Coursera, ESPN, GitHub, Google Flights, Google Maps, Google Search, Huggingface, Wolfram Alpha

태스크를 만든 방법이 흥미롭다. 3단계 Self-Instruct 방식:

Stage 1 — 시드 태스크 수집: Google Flights, Google Maps, Google Search, Booking, Wolfram Alpha 5개 사이트에 대해 Mind2Web에서 태스크를 샘플링하고 수동으로 편집해 시드 태스크 풀을 만든다.

Stage 2 — 사람 검증 생성: 시드 태스크를 in-context 예시로 주고 GPT-4 Turbo에게 사이트당 약 100개 태스크를 생성하게 한다 (20번 반복). 사람이 직접 검토하고 수정해서 Task Pool에 추가.

Stage 3 — 다양성 확보 생성: Stage 2 풀에서 다양한 예시를 샘플링해 GPT-4 Turbo에게 추가 생성. 직접 검증 후 추가. 결과: 사이트당 40개 이상 = 총 643개.

태스크 다양성은 임베딩으로 검증했다. all-mpnet-base-v2 모델로 643개 태스크를 임베딩해 유사도를 계산하면:

다양성 검증 결과 (전체 206,403 쌍)
  • 유사도 > 0.8인 쌍: 49쌍
  • 유사도 0.7~0.8인 쌍: 140쌍
  • 유사도 < 0.6인 쌍: 99.68%
거의 모든 태스크가 서로 충분히 다르다.

정답 주석도 두 가지 유형으로 나눈다:

  • "Golden" Answers (22.3%) — 안정적이고 짧은 정답이 있는 경우 (예: "파이썬 공식 문서 URL은?")
  • "Possible" Answers (77.7%) — 실시간 데이터, 열린 질문, 여러 정답이 가능한 경우 (예: "내일 서울 날씨는?")

자동 평가: GPT-4V를 심사위원으로

643개 태스크를 매번 사람이 채점하는 건 비현실적이다. WebVoyager의 혁신 중 하나는 GPT-4V를 자동 평가자로 활용한 것이다.

GPT-4V 심사위원에게 주는 입력:

  1. 태스크 지시문
  2. 에이전트 최종 응답 텍스트
  3. 마지막 k장의 스크린샷 (트레젝토리 끝부분)

k를 얼마로 설정하냐에 따라 성능이 달라진다:

k (스크린샷 수) 성공률 일관성 인간 일치율 Kappa (κ)
k=1 47.7% 75.3% 51% 0.51
k=2 55.3% 79.7% 59% 0.59
k=3 54.3% 81.3% 62% 0.62
전체 트레젝토리 58.3% 85.3% 70% 0.70

전체 트레젝토리를 줬을 때 κ=0.70으로 가장 높다. 사람끼리의 inter-annotator agreement도 κ=0.70이었다 — 즉 GPT-4V 평가자가 사람 수준의 판단력을 보였다는 것이다.

평가 온도는 0으로 설정해 결정론적 판정을 유도했다.

Google Maps가 압도적. GPT-4 All Tools 2.4%, 텍스트만 7.1% → WebVoyager 59.5%. 지도는 시각 정보가 핵심인 UI라 텍스트 기반 에이전트가 완전히 무력화되는 대표 사례다.

Allrecipes는 텍스트가 오히려 앞선다 (55.6% vs 53.3%). 레시피 사이트는 텍스트가 풍부하고 구조가 단순해서, 스크린샷이 없어도 HTML 텍스트만으로 잘 작동한다. "Since WebVoyager mostly relies on web screenshots for decision-making, dense text might not be easily recognizable from the image."

오류 분석: 실패 300개를 뜯어보면

실패한 태스크 300개를 수동으로 분류한 결과:

오류 유형 분포

44.4% — Navigation Stuck
길을 잃고 같은 행동을 반복하는 케이스. 세 가지 세부 패턴이 있다:
① 부정확한 검색어 → 무관한 결과에 압도 → 다른 결과를 계속 탐색하거나 그냥 기다림
② 작은 스크롤 영역 → 올바른 스크롤 위치를 못 찾아 쓸모없는 스크롤 반복
③ 스크롤 방향 혼동 → 위/아래 결정 못 하고 이전 실수 반복 (context clipping 부작용)

24.8% — Visual Grounding 오류
화면을 잘못 읽는 케이스:
① 인식 실패 — 발음 기호, 수식 기호 등 비표준 패턴 못 읽음
② 미세한 변화 감지 실패 — 실행은 됐는데 화면이 조금만 바뀌어서 "실패했다"고 착각
③ 인접 요소 혼동 — 달력 숫자를 Set-Label 번호로 착각하는 등 근처 요소를 헷갈림

21.8% — Hallucination
없는 정보를 만들어내거나 태스크를 대충 완료하는 케이스:
① 태스크 일부 누락 — "가장 싼" 걸 찾아야 하는데 "싼 것"으로 만족, 정렬 단계 스킵
② 유효하지만 잘못된 액션 — 엉뚱한 입력창에 텍스트 입력 (에러는 안 뜨지만 결과가 틀림)

9.0% — Prompt Misalignment
출력 형식 문제:
① 액션 없이 Thought만 출력
② `ANSWER`를 너무 일찍 써서 조기 종료 (심지어 태스크가 미완성임을 스스로 인정하면서도 종료)

가장 많은 실패(44.4%)가 "Navigation Stuck"이라는 게 의미심장하다. 사람은 막히면 전략을 바꾼다 — 검색어를 수정하거나, 뒤로 가거나, Google 액션으로 새로 시작하는 등. 에이전트는 이 메타인지적 탈출 전략이 아직 부족하다.

오픈소스 모델로는 왜 어려운가

논문이 GPT-4V만 쓴 이유를 명확히 밝힌다:

이미지 해상도 문제: LLaVA 같은 오픈소스 LMM들은 입력 이미지를 224×224 또는 336×336으로 축소한다. WebVoyager는 1024×768 스크린샷을 쓰는데, 축소하면 작은 글씨를 읽을 수 없다.

컨텍스트 길이 문제: LLaVA의 최대 컨텍스트가 4096 토큰인데, 15 step 트레젝토리를 처리하려면 7,000+ 토큰이 필요하다. 처음부터 불가능하다.

이게 2024년 초 기준 웹 에이전트가 GPT-4V에 강하게 묶여 있던 이유였다.

의의: 두 가지 기여

WebVoyager가 남긴 것은 크게 두 가지다.

첫째, 시각 정보가 결정적이라는 실증. 텍스트만 쓴 WebVoyager 40.1% → 멀티모달 WebVoyager 59.1%. +19%p. 특히 Google Maps처럼 지도·시각 UI가 핵심인 사이트에서는 텍스트 에이전트와의 격차가 압도적이다(7.1% vs 59.5%).

둘째, 실제 웹 + 결과 기반 평가의 표준 제시. 정적 스냅샷과 step 단위 비교 대신, 실시간 웹에서 최종 결과만으로 판단하는 방식. GPT-4V를 심사위원으로 활용하는 자동 평가 프로토콜은 이후 웹 에이전트 연구의 평가 기준에 직접적인 영향을 줬다.

남은 한계들:

  • 드래그, 파일 업로드 같은 복잡한 인터랙션 미지원
  • 로그인, CAPTCHA 처리 불가
  • Navigation Stuck이 가장 큰 실패 원인 — 막혔을 때 전략 전환 능력 부족
  • 오픈소스 모델로 재현 불가 (해상도 + 컨텍스트 한계)

"There is still substantial room for further improvement towards generalist agents for the web."

이 한계들을 하나씩 해결해 나간 것이 이후 SeeAct, Computer-Use, Operator로 이어진다.
