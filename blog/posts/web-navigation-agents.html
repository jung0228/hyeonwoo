Web을 탐색하는 AI Agent: 개념부터 최신 연구까지

브라우저를 직접 조작하고, 링크를 클릭하고, 폼을 채우는 AI — Web Navigation Agent의 핵심 개념과 주요 연구 흐름을 정리합니다.

Agent · Mar 3, 2026 · 12 min read

왜 Web Navigation인가

LLM이 텍스트를 잘 생성한다는 건 이제 익숙한 사실이다. 그런데 실제로 유용한 작업의 상당수는 단순히 텍스트를 뽑아내는 것이 아니라 무언가를 하는 것이다. 항공권을 검색하고, 서류를 제출하고, 가격을 비교하고, 예약을 완료하는 것. 이 모든 행위는 웹 브라우저 위에서 이루어진다.

Web Navigation Agent는 사람이 브라우저를 쓰듯이 AI가 스스로 웹을 탐색하게 만드는 연구다. 단순히 URL을 fetch해서 텍스트를 읽어오는 것과는 다르다. 동적으로 렌더링되는 페이지, 로그인 세션, 클릭 이벤트, 드롭다운, 무한 스크롤 — 실제 브라우저 환경 전체가 action space다.

핵심 질문: LLM에게 "무엇을 할지"를 알려주는 것은 이미 잘 된다. 문제는 "어떻게 실행하느냐"다. 웹이라는 복잡한 환경에서.

문제를 어렵게 만드는 것들

Web Navigation이 어려운 이유는 단순히 HTML을 파싱하는 것보다 훨씬 복잡한 환경 때문이다.

  • 거대한 observation space: 실제 웹페이지의 DOM은 수천 개의 노드를 갖는다. 그 중 어디를 봐야 하는가?
  • 긴 action sequence: 한 태스크를 완료하려면 수십 번의 클릭·타이핑이 필요할 수 있다.
  • 부분 관찰 가능성: 현재 화면에 보이지 않는 정보가 태스크 완료에 필요할 수 있다.
  • 오류 복구: 중간에 잘못된 액션을 취했을 때 스스로 감지하고 되돌아와야 한다.
  • 일반화: 훈련 때 보지 못한 새로운 웹사이트에서도 동작해야 한다.

Agent의 기본 구조

Web Navigation Agent는 일반적으로 관찰 → 계획 → 행동의 루프로 동작한다. 세 가지 요소가 핵심이다.

1. Observation — 웹을 어떻게 입력으로 받는가

브라우저 상태를 모델에게 전달하는 방법은 크게 두 가지다.

  • 텍스트 기반 (DOM / Accessibility Tree): HTML을 파싱해 텍스트 표현으로 변환한다. 토큰 수가 많아지는 게 문제지만 LLM이 직접 읽을 수 있다. Mind2Web이 이 방식을 정제했다.
  • 스크린샷 기반 (Visual): 실제 렌더링된 화면을 그대로 Vision 모델에게 넘긴다. 사람이 보는 것과 같은 정보를 사용하므로 더 자연스럽다. WebVoyager가 이 접근이다.

최근 연구들은 두 방식을 결합하거나, Accessibility Tree + 스크린샷을 함께 주는 방향으로 간다.

2. Action Space — 무엇을 할 수 있는가

브라우저에서 취할 수 있는 행동은 보통 이렇게 정의된다.

  • click(element): 특정 요소 클릭
  • type(element, text): 텍스트 입력
  • scroll(direction): 스크롤
  • navigate(url): URL 직접 이동
  • select(element, option): 드롭다운 선택
  • submit(): 폼 제출
  • done(answer): 태스크 완료 선언

어떤 요소를 지칭하느냐가 핵심 과제다. 인덱스 번호로 지칭할 수도 있고, XPath·CSS selector를 생성할 수도 있고, 좌표(x, y)로 찍을 수도 있다.

3. Planning — 어떻게 결정하는가

초기 연구들은 단순히 LLM에게 현재 상태를 주고 다음 액션을 하나 뽑는 방식이었다. 최근에는 ReAct (Reason + Act) 패턴이 표준처럼 자리 잡았다. 행동 전에 why를 먼저 생성하게 해서 추론 품질을 높이는 방식이다.

ReAct 패턴 예시:
Thought: 현재 페이지에서 "flights" 검색창을 찾아야 한다. 상단 네비게이션에 보인다.
Action: click(#search-flights-input)
Observation: 텍스트 입력창이 포커스됨.
Thought: 출발지를 입력해야 한다…

주요 연구 흐름

Mind2Web (NeurIPS 2023 Spotlight)

Mind2Web은 Web Navigation을 위한 첫 번째 대규모 벤치마크이자 Generalist Agent 연구다. 137개 웹사이트, 2,000개 이상의 태스크를 포함하는 데이터셋을 구축했다.

핵심 기여는 후보 요소 필터링 + 액션 예측의 분리다. 전체 DOM 대신 관련 요소 후보군을 먼저 추려낸 뒤, 그 중에서 액션을 선택하게 한다. 거대한 HTML을 그대로 LLM에 던지는 것보다 훨씬 효율적이다.

  • Task type: cross-website generalization (훈련 시 못 본 사이트에서 평가)
  • Observation: HTML → 정제된 텍스트 표현
  • 모델: fine-tuned LLM (GPT 계열 + 소형 모델 조합)

WebVoyager (ACL 2024)

WebVoyager는 GPT-4V를 기반으로 스크린샷을 직접 관찰로 사용하는 end-to-end 웹 에이전트다. 사람이 브라우저를 보듯 화면을 그대로 입력으로 받고, 클릭할 위치를 좌표로 출력한다.

실제 웹사이트 15개에 대해 사람이 작성한 태스크를 수행하며 평가했고, 기존 텍스트 기반 방법보다 현저히 높은 성공률을 보였다. 시각적 맥락의 중요성을 보여준 연구다 — 어디에 무엇이 있는지를 봐야 올바른 액션을 취할 수 있다.

  • Observation: 전체 페이지 스크린샷
  • Action: 화면 위 좌표 클릭 + 타이핑
  • 모델: GPT-4V (fine-tuning 없이 in-context로 동작)

Agent-E (arXiv 2024.07)

Agent-E는 단순히 새 모델을 제안하는 것을 넘어, Agentic System 설계 원칙을 체계화한 연구다. "어떻게 하면 더 강건한 웹 에이전트를 만들 수 있는가"에 대한 엔지니어링 관점의 답을 제시한다.

핵심 기여:

  • Hierarchical architecture: 고수준 Planner + 저수준 Executor로 분리. Planner는 전략을, Executor는 실제 DOM 조작을 담당한다.
  • DOM Distillation: 페이지 전체 대신 태스크와 관련 있는 DOM 요소만 추출하는 기법.
  • Change Observation: 액션 전후 DOM 변화를 관찰해 의도된 액션이 실행됐는지 확인.
  • Error recovery: 실패를 감지하고 다른 전략으로 재시도하는 루프.

WebArena 벤치마크에서 당시 SOTA였으며, 구조 설계 측면에서 후속 연구들에 많이 인용된다.

Browser Use — 오픈소스 프레임워크

Browser Use는 GitHub Star 72K를 넘긴 오픈소스 웹 에이전트 프레임워크다. 위 연구들의 아이디어를 실용적으로 구현해놓은 라이브러리라고 보면 된다.

Playwright를 기반으로 실제 Chromium 브라우저를 제어하고, LLM(OpenAI, Anthropic 등)을 두뇌로 연결한다. 구조는 단순하다.

  • 브라우저 상태를 Accessibility Tree + 스크린샷으로 캡처
  • LLM에게 현재 상태와 목표를 전달
  • LLM이 반환한 액션을 Playwright로 실행
  • 반복

연구용 프로토타입을 빠르게 만들거나, 웹 자동화 파이프라인을 LLM 기반으로 전환할 때 유용하다. 특히 pip install browser-use 한 줄로 시작할 수 있어 접근성이 높다.

주의: Browser Use는 실제 브라우저를 제어하므로 로그인 세션, 쿠키, 결제 등 실제 액션이 실행된다. 테스트 환경 분리가 중요하다.

벤치마크: 어디서 평가하는가

Web Navigation 연구에서 자주 등장하는 벤치마크들이다.

  • Mind2Web: 137개 사이트, 2K+ 태스크. Cross-website 일반화 평가.
  • WebArena: 실제처럼 동작하는 4개 웹앱(쇼핑몰, 깃허브, 포럼 등)에서 812개 태스크. 결과가 실제로 DB에 반영되어 평가 가능.
  • VisualWebArena: WebArena의 시각적 확장. 이미지·차트가 포함된 태스크.
  • WorkArena: 기업용 SaaS(ServiceNow) 환경 특화.

WebArena의 평균 성공률은 GPT-4 기반 최신 에이전트도 50%대 수준이다. 사람의 성공률(~78%)과 아직 격차가 있다.

내가 생각하는 것들

지금까지의 연구를 보면, Web Navigation은 본질적으로 정보 탐색과 순차적 의사결정이 결합된 문제다. 두 가지가 모두 잘 되어야 한다.

흥미로운 점은, 성공적인 웹 에이전트들이 공통적으로 계층적 구조를 가진다는 것이다. 한 모델이 모든 걸 하는 게 아니라, 고수준 계획과 저수준 실행을 분리한다. 이건 사람도 마찬가지 — 우리는 "항공권 예약"이라는 목표를 갖고 있지만, 실제로는 "검색창 찾기 → 클릭 → 날짜 입력 → …"으로 decompose해서 실행한다.

또 하나: 오류 감지와 복구가 생각보다 훨씬 중요하다. 완벽한 plan을 세워도 실제 실행에서 예상치 못한 일이 생긴다. 팝업이 떴다거나, 버튼 위치가 달라졌다거나. 이걸 인식하고 대처할 수 있는 에이전트가 강하다.
